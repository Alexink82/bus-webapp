# Bus Booking Web App — как устроен проект и что изменится с API

## 1. Что остаётся неизменным (логика из первого поста)

- **Роли**: пассажир, диспетчер, администратор.
- **Пассажирский сценарий**: поиск маршрута (откуда → куда → дата) → выбор рейса → форма бронирования (пассажиры, контакт, способ оплаты) → успех/трекинг заявки. Сохранённые пассажиры, «для другого человека», оффлайн-режим (PWA), FAQ.
- **Диспетчерская панель**: новые заявки / в работе / поиск / статистика; взятие в работу; смена статусов; уведомления (в перспективе — real-time). **Все диспетчеры видят все заявки** (без привязки к маршрутам).
- **Админ-панель**: статистика, логи, экспорт, управление ролями (диспетчеры/админы), при необходимости — чёрный список, архивация.
- **Интеграции**: Telegram (Web App, уведомления пассажирам), одна БД на сервере, при желании — тот же сервер, что и bus-bot.

То есть вся описанная в первом посте логика работы веб-приложения не переписывается — меняются только источники данных (откуда берутся маршруты и заявки).

## 2. Сейчас: работа «как в старом проекте» (bus-bot)

Смысл: **без внешнего API** всё работает так же, как в bus-bot — те же данные и та же БД.

### 2.1 Маршруты

- **Источник**: не API, а константы в коде (как в bus-bot `core/constants.py`).
- В webapp те же маршруты: Мозырь ↔ Москва, Гомель ↔ Мозырь (те же `id`, остановки, цены, расписание).
- Список маршрутов для фронта отдаёт бэкенд webapp’а из этих констант (единая «правда» с bus-bot).

**Итог**: маршруты не хранятся в БД webapp’а; они заданы в коде и совпадают с bus-bot.

### 2.2 База данных

- **Одна БД на сервере** — та же PostgreSQL, что использует bus-bot (тот же `DATABASE_URL`).
- **Таблицы**:
  - `bookings` — заявки в формате bus-bot (id формата `BK-…`, поля: status, route_id, from_city, to_city, date, departure, arrival, passengers, contact_phone, contact_tg_id и т.д.).
  - `bot_roles` — кто админ/диспетчер (из админ-панели бота или webapp).
- Webapp создаёт заявки в `bookings` и читает их оттуда же. Бот тоже пишет и читает эту таблицу — заявки, созданные в вебе, видны в боте и наоборот.

**Итог**: никакой отдельной «своей» БД у webapp’а нет; только общая с bus-bot.

### 2.3 Размещение

- **Тот же сервер (Render и т.п.)**: webapp — отдельный сервис (FastAPI + статика), но тот же `DATABASE_URL`, что и у bus-bot.
- Два процесса: один — бот (как сейчас), второй — webapp. БД общая.

**Итог**: одна инфраструктура, одна БД, два приложения (бот + веб).

## 3. Заглушки под будущий API

Идея: **сейчас данные берутся из констант и общей БД; когда появится внешний API — переключим источники на него**, не меняя общую логику сценариев.

### 3.1 Что может давать будущий API (предположительно)

- **Маршруты**: список маршрутов, остановки, цены, расписание.
- **Заявки**: создание/чтение/обновление статусов заявок (или только чтение, а создание по-прежнему в нашей БД по данным из API).
- **Справочники**: города, тарифы, правила отмены и т.д.

Точный контракт API пока не задан — под него и делаются заглушки.

### 3.2 Где в проекте будут заглушки

- **Слой «данные о маршрутах»**  
  Сейчас: функция/сервис возвращает маршруты из констант (как в bus-bot).  
  Потом: та же функция/сервис будет вызывать API (с кэшем/ретраями) и отдавать данные в том же формате для фронта. Внутри одна точка подмены: «источник маршрутов» (константы → API).

- **Слой «заявки»**  
  Сейчас: создание/чтение/обновление заявок идёт в нашу PostgreSQL (`bookings`).  
  Потом: при появлении API заявок — вызовы API вместо (или поверх) записи в БД. Заглушка = интерфейс «репозиторий заявок»: сейчас реализация пишет в нашу БД, потом реализация может ходить в API или комбинировать API + БД.

- **Остальное**  
  Роли, парсеры (граница, погода), уведомления в Telegram — по необходимости тоже выносим за «фасад» и при появлении API подменяем реализацию, не трогая сценарии в вебе.

**Итог**: основная логика работы (поиск → бронь → диспетчер/админ) не меняется; меняются только модули-источники данных, которые мы заготавливаем как заглушки под API.

## 4. Как изменится проект, когда появится API

Описываем направление изменений без переписывания кода сейчас.

### 4.1 Маршруты

- **Было**: список маршрутов из констант (как в bus-bot).
- **Станет**: запрос к API (например, `GET /routes` или аналог); ответ приводится к тому формату, который уже ждёт фронт (id, name, stops, base_price, schedule_days и т.д.).
- В коде остаётся один «источник маршрутов» (слой/сервис), внутри переключается с констант на API. Константы можно оставить как fallback при недоступности API.

### 4.2 Заявки

- **Было**: создание и чтение заявок в таблице `bookings` нашей БД (как в bus-bot).
- **Станет**: по контракту API — либо создание/чтение через API, либо создание по-прежнему в БД, а список/детали — из API. Синхронизация с текущей БД (если API станет основным хранилищем) решается отдельно (миграции, двойная запись на переходный период и т.д.).
- В коде остаётся один слой «работа с заявками»; внутри подменяется реализация (БД → API или API + БД).

### 4.3 Роли, парсеры, уведомления

- Логика проверки «диспетчер/админ» и откуда брать список (env + `bot_roles`) не обязана меняться; при появлении API ролей можно добавить ещё один источник за заглушкой.
- Парсеры (граница, погода) и уведомления в Telegram — при необходимости тоже выносятся за интерфейс и подменяются (например, «актуальная информация» с API вместо парсеров).

**Итог**: проект меняется точечно — подмена источников данных за заранее выделенными слоями, без смены основной логики работы в вебе.

## 5. БД: замена бота, один веб

**Ситуация**: по сути это замена бота — уходим в веб, бот больше использоваться не будет.

**Варианты:** оставить ту же PostgreSQL (бот не поднимаем) или завести отдельную БД для webapp.

**Рекомендация**: раз бот не будет использоваться — **оставить одну БД**, не заводить вторую. Webapp — единственное приложение. Когда появится API к сайтовой БД — заявки пойдут туда через API; текущая БД может остаться для истории/офлайна.

**Итог**: одна БД на сервере, один сервис (webapp). Бот отключаем.

---

## 6. Заявки и API к сайтовой БД

**Сейчас**: заявки пишутся в нашу БД. **Потом**: по API передаём данные в другую БД (сайт). Пока API нет — доступа нет. Слой «отправка заявки» сейчас пишет в нашу БД; потом будет вызывать API (при необходимости — дублировать к нам для офлайна/повторов).

## 7. Заглушки: согласованные решения

- **Два отдельных фасада**: «источник маршрутов» и «репозиторий заявок» — переключаем по отдельности.
- **Переключение автоматическое**: если API доступен — используем его, иначе fallback (без ручного конфига в проде).
- **Fallback**: API маршрутов недоступен → отдаём константы; API заявок недоступен → пишем в нашу БД и при необходимости ставим в очередь повтор в API.
- **Версии API (v1/v2)**: это касается только бэкенда и контракта с внешним API. Для пользователя (пассажир, диспетчер в браузере/Telegram) смена версии API не видна — он по-прежнему видит тот же интерфейс и те же данные. Переход на новый API = на нашей стороне подключаем адаптер под v2, переключаем источник (авто или конфиг), при необходимости оставляем старый адаптер как fallback. Пользователю ничего «переходить» не нужно; дискомфорта нет, если адаптер отдаёт фронту тот же формат данных.

## 8. Формат номера заявки (согласовано)

**Требования**: короткий буквенно-цифровой 8–10 символов (A–Z, 0–9), удобно диктовать и вводить; желательно иметь в номере дату.

**Формат**: дата + короткий суффикс.

- **Дата**: 6 символов — `YYMMDD` (год, месяц, день), например `240615` = 15 июня 2024.
- **Суффикс**: 4 буквенно-цифровых символа (A–Z, 0–9), например `A7K2`.

**Пример номера**: `240615A7K2` (10 символов). Удобно читать по телефону, из номера видна дата создания заявки. **Алфавит суффикса** (без неоднозначных при диктовке): A–Z кроме O, цифры 2–9 (исключаем 0, 1, O, I, L) — 32 символа; проверка уникальности в БД при записи; при коллизии — повтор генерации.

## 9. Рекомендации по аналогам (2025–2026)

Перенос из анализа аналогичных решений (подробнее см. `docs/ANALOGUES_AND_BEST_PRACTICES_2026.md`).

| Область | Рекомендация |
|--------|---------------|
| **Источники данных** | Два фасада (маршруты, заявки), авто переключение API / fallback; при необходимости circuit breaker после серии ошибок (1–2 мин пауза). |
| **Очередь повторов** | Таблица в нашей БД (`api_sync_queue`), фоновый воркер с backoff, лимит попыток (например 10), при срыве — лог/уведомление админу. |
| **PWA / офлайн** | Workbox или те же стратегии: cache-first статика, stale-while-revalidate маршруты, отправка заявок при наличии сети или через Background Sync с индикацией «отправится при подключении». |
| **Номер заявки** | YYMMDD + 4 символа; алфавит без 0, 1, O, I, L; проверка уникальности при записи. |
| **Версии API** | Адаптеры внутри фасада; для пользователя невидимо. |
| **Масштаб** | Один бэкенд, одна БД, без микросервисов; rate limiting и логирование. |

## 10. Приложение как вебка в Telegram: скорость и надёжность

**Контекст**: приложение в основном запускается как **Web App внутри Telegram** (вебка в клиенте). Нужны быстрая загрузка и максимальная устойчивость к сбоям.

### 10.1 Быстрая работа

- **Первый экран за < 2–3 сек на 3G**  
  - Минимизировать объём первого ответа: только критичный HTML/CSS/JS для первого экрана (поиск или лендинг).  
  - Скрипт `telegram-web-app.js` подключать в `<head>` первым, до остальных скриптов.  
  - Ориентир по практике Mini App: неплохо держать **неразжатый JS первого экрана в районе 1 MB** (на 3G Fast это даёт приемлемое время загрузки).

- **Статика**  
  - Минимум внешних шрифтов; системные шрифты или один веб-шрифт с подмножеством символов.  
  - Картинки/иконки — по возможности SVG или сжатые WebP, ленивая подгрузка ниже первого экрана.  
  - Кэширование статики (заголовки Cache-Control, длинный TTL для хешированных имён файлов).  
  - При использовании сборки (Vite/webpack) — code splitting: подгружать страницы бронирования, профиля, диспетчера по мере перехода, а не одним бандлом.

- **API**  
  - Список маршрутов — один лёгкий запрос, ответ кэшировать на клиенте (и в SW) на 5–10 минут.  
  - Не блокировать первый экран запросами: показывать форму поиска сразу, подставлять маршруты по мере прихода ответа.  
  - Таймауты запросов — разумные (например 10–15 с); при таймауте — понятное сообщение и кнопка «Повторить».

- **Сеть**  
  - HTTPS обязательно (WebView в Telegram требует).  
  - Сервер по возможности ближе к основной аудитории (гео или CDN для статики).  
  - Учитывать мобильные сети: не делать лишних тяжёлых запросов при открытии, повтор при временных ошибках (retry 1–2 раза с задержкой).

### 10.2 Максимально без сбоев

- **WebView может закрыться в любой момент**  
  - Не хранить единственную «правду» только во фронте: критичные данные (созданная заявка, выбранные пассажиры перед отправкой) при успешном ответе сервера считать сохранёнными на бэкенде.  
  - Долгие операции (создание заявки, оплата) делать идемпотентными где возможно: повторная отправка с тем же idempotency key не создаёт дубликат.  
  - При успешном ответе «заявка создана» показывать номер заявки и при необходимости сохранять в `localStorage`/sessionStorage для восстановления экрана «успех» при повторном открытии по ссылке.

- **Ошибки и повторные попытки**  
  - Все запросы к API обрабатывать с retry при сетевых/временных ошибках (например 1–2 повтора с паузой 1–2 с).  
  - Понятные сообщения пользователю: «Нет связи», «Сервер временно недоступен. Попробуйте через минуту» и кнопка «Повторить».  
  - Не показывать сырые технические ошибки; логировать их на бэкенде для разбора.

- **Валидация и безопасность**  
  - **initData** из Telegram всегда проверять на бэкенде (криптографическая проверка подписи). Не доверять данным с фронта для идентификации пользователя.  
  - Ввод пользователя валидировать и на клиенте (быстрый фидбек), и на сервере (источник истины).

- **Вертикальные свайпы и скролл**  
  - В WebView на мобильных случайный свайп может интерпретироваться как закрытие или «подтягивание». Рекомендуется: высота контента больше высоты видимой области; при необходимости использовать методы управления свайпами (например `enableVerticalSwipes` / `disableVerticalSwipes`, Bot API 7.7+), чтобы не было случайных закрытий при скролле формы.

- **Сессия и состояние**  
  - Не полагаться на долгоживущую сессию: при повторном открытии вебки пользователь может прийти «с нуля». Важные данные (номер заявки, «мои заявки») подтягивать с сервера по `user_id` из проверенного `initData`.  
  - Сохранять прогресс формы в `localStorage` (черновик брони), чтобы при случайном закрытии можно было восстановить ввод.

- **Мониторинг**  
  - Логировать на бэкенде ошибки и таймауты; при желании — метрики времени ответа ключевых эндпоинтов. Это поможет быстро находить сбои и «тормоза» при работе из Telegram.

**Итог**: быстрая работа — за счёт малого первого ответа, кэша, одного запроса маршрутов и неблокирующего UI; надёжность — за счёт бэкенда как источника истины, идемпотентности, retry, проверки initData и аккуратной работы с WebView (скролл/свайпы, восстановление состояния).

## 11. Что дальше решить / описать

- **Очередь повторов**: см. раздел 9 (таблица в БД, воркер, backoff, лимит 10, уведомление админу).
- **Неоднозначные символы в номере**: см. раздел 8 — алфавит без 0, 1, O, I, L.
- **Авто-переключение на API**: health-check или лёгкий запрос раз в N минут; при каждом запросе можно пробовать API и при ошибке fallback; circuit breaker на 1–2 мин после серии ошибок.
- **Порядок внедрения**: сначала один фасад (маршруты), потом заявки, или оба при появлении API.
- **Telegram**: при реализации учесть раздел 10 (первый экран, кэш, retry, проверка initData, сохранение черновика, управление свайпами при необходимости).
- **Дизайн**: лаконичный интерфейс, спокойные цвета, яркие акценты, скругления, плавные анимации — см. **`docs/DESIGN_GUIDELINES.md`**.

## 12. Вопросы для согласования (архив)

Чтобы мы с тобой друг друга поняли и потом не переделывать лишнее:

1. **Маршруты**  
   Подтверждаешь, что сейчас маршруты в webapp должны быть **строго как в bus-bot** (те же id, города, цены), и что при появлении API мы просто переключим источник на API, сохранив формат ответа для фронта?

2. **База данных**  
   Верно ли, что webapp **обязательно** использует ту же PostgreSQL, что и бот (тот же сервер/инстанс), и отдельную БД для webapp заводить не планируем?

3. **Заявки**  
   Сейчас заявки создаются и в боте, и в вебе — обе стороны пишут в одну таблицу `bookings`. Так и должно оставаться до появления API? И когда API появится — он будет главным хранилищем заявок (бот и веб ходят в API) или API только отдаёт данные, а пишем по-прежнему в нашу БД?

4. **Заглушки**  
   Устраивает ли такой вариант: в коде явно есть «слой данных» (маршруты, заявки), внутри которого сейчас — работа с константами и БД; позже мы подставим туда вызовы API, не меняя вызовы этого слоя из основной логики?

5. **Диспетчеры и маршруты**  
   В bus-bot диспетчеры не привязаны к конкретным маршрутам — любой диспетчер видит все новые заявки. В первом посте была идея «заявки по маршрутам диспетчера». Оставляем пока «все заявки всем диспетчерам» как в bus-bot или сразу закладываем фильтр по маршрутам (например, из конфига/БД)?

6. **Формат ID заявки**  
   Сейчас в bus-bot id вида `BK-150624-143052`. Нужно ли webapp’у строго придерживаться этого формата при создании заявок из веба?

Ответы на эти пункты достаточно, чтобы дальше описывать или менять код так, чтобы всё совпадало с твоими ожиданиями и с логикой «пока как в старом проекте, потом — под API».

Подробный разбор аналогов и практик 2025–2026 см. в **`docs/ANALOGUES_AND_BEST_PRACTICES_2026.md`**.

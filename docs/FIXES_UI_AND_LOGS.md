## 2. Логи сервера: почему 403 по диспетчерской

В логах видны ответы:

```text
GET /api/dispatcher/bookings -> 403
GET /api/dispatcher/stats -> 403
```

Это **не ошибка приложения**, а корректная работа проверки прав:

- Запросы идут с одного и того же IP (208.54.104.184), с которым заходят в админку и на диспетчерскую.
- Роль диспетчера определяется по **Telegram User ID** (заголовок `X-Telegram-User-Id`), а не по IP.
- Список диспетчеров задаётся в переменной окружения **DISPATCHER_IDS** на Render (и/или в БД через раздел «Диспетчеры» в админке).
- Если Telegram ID текущего пользователя **не входит** в DISPATCHER_IDS (и не добавлен в БД как диспетчер), API возвращает **403** и сообщение вроде «Нет доступа (вы не диспетчер)».

Итог: 403 в логах означают, что пользователь открыл панель диспетчера, но его ID не в списке диспетчеров.

### Что сделать, чтобы 403 не появлялись для этого пользователя

1. **Добавить пользователя в диспетчеры**
   - В **Render** → сервис бэкенда → Environment: проверить/задать `DISPATCHER_IDS` (через запятую Telegram ID диспетчеров), **или**
   - В **админке** приложения: раздел «Диспетчеры» → добавить Telegram ID нужного пользователя (сохраняется в БД).

2. **Не открывать панель диспетчера без прав (опционально)**
   - Можно до перехода на `dispatcher.html` вызывать `GET /api/user/roles` и, если `is_dispatcher === false`, сразу перенаправлять на `index.html` и не вызывать `/api/dispatcher/*`. Тогда в логах не будет лишних 403 при заходе «не диспетчера» на страницу диспетчера. Текущее поведение (показать «Нет доступа (вы не диспетчер)») при этом можно сохранить для прямых ссылок.

### Остальные записи в логах

- **POST /api/bookings -> 200** (3555 ms и 726 ms) — успешное создание заявок; большая длительность может быть из-за сети или нагрузки на БД.
- **GET /api/health -> 200** — проверки живости (в т.ч. внешние мониторы).
- **GET /api/admin/* -> 200** — успешные запросы админки.
- **GET /api/user/roles, /api/user/dashboard -> 200** — успешно.

Отдельных «ошибок» в присланном фрагменте логов нет; требуют внимания только 403 по диспетчерской и при необходимости — добавление пользователя в DISPATCHER_IDS или перенаправление с `dispatcher.html` при отсутствии роли диспетчера.

**Сделано:** при открытии страницы диспетчера сначала запрашиваются роли (`GET /api/user/roles`); если пользователь не диспетчер и не админ — выполняется редирект на `index.html`, запросы к `/api/dispatcher/*` не отправляются, 403 в логах не появляются.

## 3. Сборка и деплой фронтенда

- Источник фронтенда остаётся в `webapp/`.
- Production-статика теперь обслуживается из `dist/`, а backend сначала ищет именно эту папку.
- Dockerfile собирает фронтенд автоматически в отдельном stage и подставляет готовый `dist/` в итоговый образ.
- Команда `npm run copy-dist` больше ничего не перезаписывает и оставлена только для обратной совместимости.

**Деплой:** после правок во фронтенде достаточно `npm run build`; вручную копировать `dist/` в `webapp/` больше не нужно.

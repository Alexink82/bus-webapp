## 2) Карта взаимосвязей между файлами

- `webapp/js/api.js` формирует заголовки `X-Telegram-User-Id` и `X-Telegram-Init-Data`, а backend проверяет их в `backend/api/auth_deps.py`.
- `backend/main.py` подключает все роутеры и раздаёт `webapp/` как статический фронтенд.
- `backend/api/booking.py` использует маршруты и цены из `core.constants`, валидацию, роли, уведомления.
- `backend/api/dispatcher.py` и `backend/api/admin.py` делят модель `Booking` и слой ролей.
- `backend/api/payment.py` обновляет заявку и таблицу `WebPayTransaction`.
- `backend/api/websocket.py` зависит от Telegram-валидации и диспетчерских прав.

## 3) Главные сильные стороны

1. Единая ролевая модель (env + БД роли) с повторным использованием в API.
2. Защита Telegram initData при наличии `BOT_TOKEN`.
3. Базовая защита от спама через rate limit middleware.
4. Покрыт минимальный smoke-набор API-тестов.
5. Аккуратная деградация при ошибках нотификаций/парсеров.

## 4) Критичные/высокие риски (найдено)

### [CRITICAL] Утечка деталей заявки для любого диспетчера
В `GET /api/bookings/{booking_id}` проверяется факт «пользователь — диспетчер», но не проверяется, что заявка относится к маршрутам этого диспетчера. **Исправлено:** добавлена проверка `b.route_id in route_ids` (или полный доступ при пустом списке маршрутов диспетчера).

### [HIGH] WebSocket broadcast отправляет новые заявки всем диспетчерам
**Исправлено:** рассылка фильтруется по маршрутам диспетчера (при подключении сохраняются route_ids, broadcast только для подходящих).

### [HIGH] Сумма платежа доверяется клиенту
**Исправлено:** в `/api/payment/create` сумма берётся из брони (`Booking.price_total`), а не из тела запроса.

### [MEDIUM] Тестовое покрытие матрицы прав
Рекомендация: добавить параметризованные API-тесты для access-control (владелец/диспетчер/админ/чужой).

## 5) Потенциальные будущие ошибки

1. Рост памяти в in-memory rate-limit при большом числе уникальных IP.
2. Строковые даты/время в `Booking` усложняют аналитику и timezone.
3. Смешанная модель статусов оплаты/заказа (`paid_at` vs `status`).
4. Отсутствие идемпотентности create_booking при сетевых ретраях.
5. Health-check без проверки доступности БД.

---

## 6) Современные подходы (что внедрить)

1. Policy-based authorization (RBAC/ABAC): `can_view_booking`, `can_cancel_booking` и т.д.
2. Idempotency-Key для write-операций.
3. Typed DTO + state-machine для переходов статусов.
4. Outbox pattern для уведомлений/WebSocket.
5. Метрики, request-id, structured logging, SLO.
6. CSP, security headers, ужесточение CORS в проде.

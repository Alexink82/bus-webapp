# Аудит логики ролей (клиент / диспетчер / админ) и сравнение с аналогами

Дата: 2026-03-12

## 1) Что уже удобно и работает хорошо

### Клиент
- Простой сценарий: поиск рейса → выбор → пошаговое бронирование.
- Есть фильтрация уже ушедших отправлений на выбранную дату.
- Есть сохранённые пассажиры, профиль, FAQ.
- Есть локализованные ошибки и понятные сообщения по валидации.

### Диспетчер
- Есть разделение по вкладкам (новые / в работе / поиск / статистика).
- Есть быстрые статусы и "взять в работу".
- Есть realtime-канал через WebSocket.

### Админ
- Есть статистика, экспорт, журналы, управление ролями.
- Роли комбинируются из env + БД (операционно удобно).

---

## 2) Что сейчас НЕудобно (по ролям)

### Клиент: UX gaps
1. Нет быстрых действий "повторить предыдущий маршрут" / "любимые маршруты".
2. Нет фиксированной панели прогресса заявки после создания (что дальше, кто обрабатывает, ETA).
3. Нет in-app push статусов в интерфейсе (только опора на Telegram-сообщения).

### Диспетчер: UX + операционные gaps
1. Нет группировки по приоритету (например "просроченные" заявки отдельно).
2. Нет SLA-таймера в карточке заявки (сколько минут ждёт).
3. Нет массовых действий (bulk assign / bulk status change).
4. Поиск бедный: нет фильтров по маршруту/дате/оплате/источнику.

### Админ: управленческие gaps
1. Нет обзорного KPI-дашборда по периодам (день/неделя/месяц в одном экране).
2. Нет аудита изменений ролей (кто, когда назначил/снял права).
3. Нет health-виджета "операционное состояние" (последний старт, ошибки за N минут, глубина очереди заявок).
4. Нет UX-ограничителей для критичных действий (двухэтапные подтверждения/undo).

---

## 3) Сравнение с аналогичными приложениями (best practices)

### Что обычно есть в зрелых bus/ticketing webapps
1. **Клиент:** one-tap повтор брони, карточка "моя активная заявка", push/real-time обновления статуса в приложении, цена/условия/багаж в одном блоке перед оплатой.
2. **Диспетчер:** очередь с приоритетами, SLA/таймер ожидания, быстрые шаблоны действий, клавиатурные шорткаты/массовые операции.
3. **Админ:** модуль аналитики с KPI и воронкой, полноценный audit trail, алерты по сбоям и деградации, feature flags.

---

## 4) Что добавить в проект в первую очередь

### P0 (высокий эффект)
1. **Dispatcher SLA panel**: "Новые" + "Просроченные > 15 мин".
2. **Клиентский блок "Моя активная заявка"** на главной/профиле.
3. **Админский audit log ролей** (назначение/удаление прав).
4. **Расширенные фильтры диспетчера** (маршрут, дата, статус оплаты).

### P1
1. Bulk actions для диспетчера (массовые статусы).
2. Шаблоны сообщений клиенту для частых статусов.
3. KPI-переключатель периодов в админке.
4. In-app timeline заявки для клиента.

### P2
1. Рекомендательная логика маршрутов (быстрые маршруты/частые направления).
2. Персональные "избранные" точки отправления/прибытия.
3. Продуктовая аналитика по drop-off шагов бронирования.

---

## 5) Предложение по перестройке меню и блоков

### Для клиента (нижняя навигация)
- Бронирование | Мои заявки | Профиль | Поддержка/FAQ

### Для диспетчера
- Очередь (Новые + Просроченные) | В работе | Завершённые | Аналитика смены

### Для админа
- Dashboard | Операции | Пользователи/Роли | Логи/Аудит | Экспорт

---

## 6) Итог

- По клиентскому пути проект уже "рабочий прод-MVP".
- По диспетчерскому и админскому контуру — сильная база, но до уровня "операционно зрелой системы" не хватает приоритизации очередей, SLA-индикаторов, аудита и расширенной аналитики.
- Рекомендуемая стратегия: 2 коротких спринта (P0 + P1), затем метрики и продуктовая оптимизация (P2).

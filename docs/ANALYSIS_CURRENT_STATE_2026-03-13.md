## 1. Общий вид приложения и админка

**Что планировалось (по документации):** Единый визуальный стиль для всего webapp: общий theme-layer (`telegram-theme.css`), одинаковые цвета/контрасты, кнопки и компановка (см. `docs/UI_STYLE_SYSTEM_AUDIT.md`, работа с Codex).

**Текущее состояние:**
- Основное приложение (index, booking, profile, success, faq) использует `main.css` + `telegram-theme.css`, общие переменные (`--bg`, `--surface`, `--primary`, `--accent`), нижняя навигация, единые tap-target и отступы.
- **Админка** (`admin.html`) подключена к **`design-system.css`** и `telegram-theme.css`, разметка и классы другие: `admin-header`, `admin-tabs`, `admin-panel`, `btn--primary`, `btn--secondary`, `form-group`, `input` из design-system. Цвета и типографика задаются в `design-system.css` (другая палитра и токены: `--color-bg`, `--color-surface` и т.д.).
- В итоге админка визуально «выбивается»: другая шапка, другие кнопки, другие отступы и карточки. Чек-лист из `UI_STYLE_SYSTEM_AUDIT.md` («Одинаковый цвет фона/текста на index, booking, profile, success, faq, dispatcher, admin») не выполнен для admin.

**Что не работает:** Единый стиль между основным приложением и админкой. Ранее заложенная унификация (общий theme, расстановка кнопок) к админке полностью не применена — она остаётся на design-system.

**Файлы:** `webapp/admin.html` (подключение CSS), `webapp/css/design-system.css`, `webapp/css/main.css`, `webapp/css/dispatcher.css` (диспетчер ближе к main).

## 5. Кнопка «Перенести дату» и правила отмены/переноса по времени отправления

**Бизнес-правила (подтверждённая логика):**
- Отсчёт всегда от **времени отправления** рейса.
- **Международные рейсы** (`route.type === "international"`): если до отправления **менее 2 часов** — отмена и перенос даты возможны **только через диспетчера** (обращение в поддержку).
- **Междугородние/внутренние рейсы** (`route.type === "local"`): если до отправления **менее 15 минут** — отмена и перенос даты возможны **только через диспетчера**.
- Если правило срабатывает, пользователю показывается сообщение о необходимости обратиться к диспетчеру; самостоятельная отмена или переход к новому бронированию без участия диспетчера в этом окне недоступны.

**Текущее состояние:**
- В профиле «Перенести дату» в модалке «Подробнее» — просто ссылка на `index.html?from=...&to=...`. Текущая заявка не отменяется, проверки времени и типа маршрута нет.
- На бэкенде в `cancel_booking` для владельца при статусе не `new` используется **единый порог 2 часа** для всех заявок; различие международный (2 ч) / междугородний (15 мин) **не реализовано**. Код `cancel_only_via_dispatcher` возвращается при отправлении менее чем через 2 ч. В api.js есть сообщение про «2 часа», без упоминания 15 минут для местных.

**Что не работает:** (1) На бэкенде нет учёта типа маршрута: для `local` должен быть порог 15 минут, для `international` — 2 часа. (2) «Перенести дату» не проверяет ни время, ни тип маршрута и не показывает подсказку про диспетчера при срабатывании правила. (3) Текст для пользователя должен различаться: «менее 2 ч» для международных, «менее 15 мин» для междугородних.

**Файлы:** `backend/api/booking.py` (cancel_booking — ввести порог по route_type), `webapp/js/profile.js` (showBookingDetailsModal, логика «Перенести дату»), `webapp/js/api.js` (сообщения cancel_only_via_dispatcher / аналог для 15 мин).

## 6. Блок «Актуальная информация»: парсинг границы, погода, геолокация

**Что ожидается (по запросу):** Убрать парсинг границы; показывать актуальную информацию по погоде; определять местоположение пассажира из Telegram; показывать погоду в точке нахождения и в конечных точках по всем маршрутам; по «Подробнее» — расширенный вывод.

**Текущее состояние:**
- На главной (index) секция «Актуальная информация»: два блока «Загрузка...» (`#borderInfo`, `#weatherInfo`) и ссылка на канал. Данные запрашиваются с `GET /api/news`.
- Бэкенд `GET /api/news` отдаёт кэш из таблицы `cached_data`: ключи `border_status` и `weather`. Данные заполняются фоновой задачей `_update_cache()` в `parsers/scheduler.py`: парсинг границы (`parsers/border.py`), погода (`parsers/weather.py`). При пустом кэше возвращаются заглушки.
- На фронте в index после получения `data` выводится: `data.border.message_ru` и `data.weather.city`, `temp`, `description_ru`. Геолокация из Telegram не запрашивается, погода по маршрутам и «подробнее» не реализованы.

**Что не работает:**
- Парсинг границы по-прежнему в коде и в кэше; его нужно отключить/убрать по требованию.
- Погода одна (один город из кэша), не привязана к местоположению пользователя и не показывается по конечным точкам маршрутов.
- Определение местоположения из Telegram (WebApp геолокация) не используется.
- Блока «Подробнее» с расширенной погодой по маршрутам нет.

**Файлы:** `webapp/index.html` (секция `#actualInfo`, fetch `/api/news`), `backend/api/faq.py` (эндпоинт `/api/news`), `backend/parsers/scheduler.py`, `backend/parsers/border.py`, `backend/parsers/weather.py`; при доработке — новый API/логика погоды по геолокации и по городам маршрутов.

## Как не ломать рабочий код при следующих правках

1. **Коммиты:** Перед началом работ по пунктам 1–6 сделать коммит с описанием текущего рабочего состояния (маршруты, логи, бронирование). Крупные изменения (стили админки, структура профиля, success/назад, блок актуальной информации) вести отдельными коммитами с коротким описанием, чтобы при необходимости откатиться по файлу или коммиту.
2. **Теги:** После стабилизации можно выставить тег, например `v1.0-working-baseline`, на коммит, где маршруты и бронирование стабильно работают.
3. **Чек-лист перед релизом:** В `docs/testing/MANUAL_TEST_CHECKLIST.md` (или отдельный файл) зафиксировать минимальный сценарий: главная → выбор маршрута/даты/времени → бронирование → успех → логи с API start/done в Render. При любых изменениях в booking.js, main.py, routes, api/bookings — прогонять этот сценарий.
4. **Не трогать без необходимости:** Участки кода, отвечающие за загрузку маршрутов (`booking.js` fetch и обработка ответа, `api/routes.py`), логирование в main.py и CMD в Dockerfile — не рефакторить «заодно»; менять только при явной задаче, с тестами/ручной проверкой.

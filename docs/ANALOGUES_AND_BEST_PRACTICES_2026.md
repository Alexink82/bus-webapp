# Аналогичные решения и лучшие практики (2025–2026)

Краткий разбор подходов в похожих приложениях (бронирование билетов, транспорт, PWA, мульти-источники данных) и предложения для Bus Booking Web App.

---

## 1. Архитектура данных: фасад + адаптеры, авто fallback

**Что используют в индустрии**

- **Adapter + Facade**: единая нормализованная модель данных для приложения, за ней — адаптеры под каждого провайдера (константы, API v1, API v2). Менять/добавлять провайдеров можно без переписывания бизнес-логики.
- **Авто failover**: при недоступности основного источника — переключение на fallback по health-check или по ошибкам запросов; retry с backoff и лимитом попыток.
- **Backend for Frontend (BFF)**: отдельный слой под конкретный клиент (веб/Telegram), который собирает данные из нескольких источников и отдаёт один API. Для нашего случая наш FastAPI уже играет роль BFF: фронт ходит только в него, а он решает, брать маршруты из констант или API.

**Предложение под наш проект**

- Два фасада: «маршруты» и «заявки» — уже согласовано.
- Переключение **автоматическое**: при старте или по расписанию (раз в N минут) проверяем доступность внешнего API (health или лёгкий запрос). Если ок — используем API, иначе константы/наша БД. При каждом запросе можно дополнительно пробовать API и при ошибке сразу fallback, чтобы не ждать следующего цикла.
- **Circuit breaker**: после нескольких подряд ошибок к API считать его «сломанным» на 1–2 минуты, не долбить запросами, использовать только fallback. После паузы — снова пробовать.
- Для масштаба одного региона и одного внешнего API тяжёлые очереди (Kafka и т.п.) не обязательны; достаточно таблицы в нашей БД для очереди повторов.

## 2. Очередь повторов при недоступности API заявок

**Паттерны**

- Очередь мутаций (создание/обновление заявок): при ошибке API пишем в локальную очередь, фоновый воркер периодически повторяет с ограничением частоты и лимитом попыток.
- Retry с экспоненциальным backoff (1 с, 2 с, 4 с…) и максимум 5–10 попыток; после — помечать как «требует ручной проверки» и при желании уведомлять админа.

**Предложение**

- **Хранение**: таблица в нашей PostgreSQL, например `api_sync_queue`: id заявки (или payload), тип операции (create/update), попытки, next_retry_at, статус (pending/failed/success).
- **Воркер**: фоновая задача (отдельный процесс или периодическая задача в том же приложении) раз в 30–60 секунд забирает записи с `next_retry_at <= now()` и повторяет вызов API; при успехе — помечать success, при ошибке — увеличивать счётчик и переносить next_retry_at с backoff.
- **Лимит**: например 10 попыток; после — статус `failed`, опционально уведомление админу (логирование/email/Telegram).
- Пользователь при создании заявки сразу получает ответ «заявка создана» (сохранена у нас); синхронизация с сайтовой БД идёт в фоне. В UI при желании можно показывать «синхронизация с системой…» и потом «синхронизировано» или «ошибка синхронизации, повторим автоматически».

## 3. PWA и офлайн (2025–2026)

**Тренды**

- Service Worker — основа: кэш статики и опционально данных, background sync для отложенной отправки действий.
- **Workbox** — стандартная библиотека (используют многие PWA): прекеш, маршрутизация запросов, стратегии кэширования (cache-first для статики, network-first или stale-while-revalidate для API).
- Стратегии по типу контента: статика — cache-first; список маршрутов — stale-while-revalidate (сначала кэш, в фоне обновление); создание заявки — network-only с очередью при офлайне (background sync).
- Для бронирований с ограниченной вместимостью в офлайне обычно не создают «окончательную» бронь без сервера, а сохраняют черновик и отправляют при появлении сети, с риском отказа при нехватке мест.

**Предложение**

- Использовать **Workbox** (или его принципы), если собираем фронт через сборщик (Vite/webpack и т.д.): меньше ручных багов в Service Worker, готовые стратегии.
- **Офлайн**: кэшировать статику и список маршрутов (из нашего API); форму бронирования показывать, но отправку делать только при наличии сети или через Background Sync (очередь в SW). При офлайне явно показывать: «нет связи, заявка будет отправлена при подключении».
- Ограничить офлайн-сценарий: просмотр маршрутов и заполнение формы — да; «гарантированная бронь без сети» — нет, только отложенная отправка с возможным отказом на стороне API.

## 4. Номер заявки: уточнение под лучшие практики

**Практика индустрии**

- Короткие коды (6–10 символов), удобные для диктовки и ввода.
- Исключать неоднозначные символы: **0/O**, **1/I/L** — при наборе по телефону и при переписке меньше ошибок.
- Часто только буквы (A–Z без O) и цифры (1–9 без 0); либо алфавит из 32–34 символов.
- Уникальность через БД/индекс; при коллизии — повторная генерация.

**Уже согласовано**

- Формат: дата (YYMMDD) + короткий суффикс, 8–10 символов, желательно с датой.

**Предложение**

- **Формат**: `YYMMDD` + 4 символа суффикса = 10 символов. Пример: `240615A7K2`.
- **Алфавит суффикса**: A–Z кроме O, цифры 2–9 (исключаем 0, 1, O, I, L). Получается 32 символа; 32^4 ≈ 1 млн комбинаций в день — для одного региона достаточно.
- **Генерация**: случайные 4 символа из этого алфавита; перед записью проверка уникальности в БД за разумное окно (например за текущий день); при коллизии — новая попытка (1–2 раза).
- Так номер остаётся коротким, с датой и удобным для диктовки.

## 5. Масштаб и упрощения под наш кейс

Для одного региона, одного оператора и одного будущего API не нужны:

- Микросервисы и Kafka — достаточно одного приложения (FastAPI) и одной БД.
- Шардирование БД — не нужно на старте.
- Отдельный API Gateway с сложным роутингом — наш бэкенд сам является «шлюзом» к константам и к внешнему API.

Имеет смысл:

- Один бэкенд (BFF), два фасада (маршруты, заявки), авто fallback и простая очередь повторов в нашей БД.
- Rate limiting на уровне приложения (например 10–60 запросов в минуту на пользователя), чтобы защититься от злоупотреблений.
- Логирование и при желании метрики (время ответа, доля fallback, ошибки API) для мониторинга.

## 6. Итоговые рекомендации (сводка)

| Область | Рекомендация |
|--------|---------------|
| **Источники данных** | Два фасада (маршруты, заявки), авто переключение API / fallback, при необходимости простой circuit breaker после серии ошибок. |
| **Очередь повторов** | Таблица в нашей БД (`api_sync_queue`), фоновый воркер с backoff, лимит попыток (например 10), при срыве — лог/уведомление админу. |
| **PWA / офлайн** | Workbox или те же стратегии вручную: cache-first статика, stale-while-revalidate маршруты, отправка заявок при наличии сети или через Background Sync с явной индикацией «отправится при подключении». |
| **Номер заявки** | YYMMDD + 4 символа; алфавит без 0, 1, O, I, L; проверка уникальности при записи. |
| **Версии API** | Адаптеры внутри фасада; для пользователя невидимо. |
| **Масштаб** | Один бэкенд, одна БД, без микросервисов и шардирования; rate limiting и логирование. |

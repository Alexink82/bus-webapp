### 2.2 Инициализация и закрытие

| Функция из ТЗ | У вас | Рекомендация |
|---------------|--------|--------------|
| `tg.ready()` | **Есть** — вызывается на index, success, faq, admin и т.д. | Оставить как есть |
| `tg.expand()` | **Нет** | **Взять в проект:** один вызов при загрузке (например в auth.js или в одном общем скрипте), чтобы WebApp открывался на весь экран |
| `tg.close()` | **Нет** | **Взять в проект:** кнопка «Закрыть» в шапке (особенно на success/главной), по клику вызывать `Telegram.WebApp.close()` |
| `goBack()` (history.back / close) | **Нет** (есть только ссылки по страницам) | По желанию: на экранах «вглубь» (например бронирование) — кнопка «Назад» с `history.back()`, иначе `close()` |

**Что можно добавить в проект:**

- В **auth.js** (или в общем init-скрипте) после проверки `Telegram.WebApp`: вызвать `tg.expand()`.
- Небольшой блок в шапке (или в общем CSS/HTML): кнопка «Закрыть» с вызовом `Telegram.WebApp.close()` — на главной и на success этого достаточно; при желании дублировать на остальных страницах.

**Сделано в проекте:** в auth.js добавлены `tg.ready()` и `tg.expand()`. На index.html и success.html добавлена кнопка «✕» в шапке с вызовом `Telegram.WebApp.close()`; в main.css добавлен класс `.tg-close-btn`. Кнопка скрывается, если WebApp недоступен (открытие в обычном браузере).

Дополнительно: кнопка «Открыть заявки» в уведомлениях (reply_markup с web_app при заданном WEBAPP_URL); парсинг start_param на фронте и передача в API для логирования в create_booking.

## 3. Модуль: персональные уведомления

### 3.1 Отправка через Bot API

| ТЗ | У вас |
|----|--------|

| Типы: LOVE_EGG, BREEDING_COMPLETE, DAILY_BONUS | Типы: создание заявки, смена статуса заявки |

**Реализовано:** да, у вас уже есть персональные уведомления и единая точка отправки. **Переносить Node-сервис не нужно.**

### 3.2 Кнопка «Открыть приложение» в сообщении

- **ТЗ:** в сообщениях используется `reply_markup` с `inline_keyboard` и кнопкой с `web_app: { url }`.
- **У вас:** текст с заявкой/статусом без кнопки.

**Что можно взять в проект:** добавить в уведомления (создание заявки, смена статуса) кнопку «Открыть заявки» / «В приложение» с `web_app: { url: WEBAPP_URL }`. В Python это делается через поле `reply_markup` в `sendMessage` (см. документацию Telegram Bot API). Так пользователь из чата сразу перейдёт в WebApp.

**Сделано:** в `notify_booking_created` и `notify_booking_status` добавлен `reply_markup` с кнопкой «📋 Открыть заявки», если в конфиге задан валидный `WEBAPP_URL` (начинается с http). При пустом или невалидном URL кнопка не добавляется — поведение как раньше, без крашей.

### 3.3 Массовая рассылка и rate limit

- **ТЗ:** цикл по пользователям, задержка ~35 ms между сообщениями (лимит Telegram).
- **У вас:** уведомления точечные (одна заявка — один получатель), массовой рассылки нет.

**Вывод:** для текущей логики ничего менять не нужно. Если позже появится массовая рассылка — добавить задержку между вызовами `send_telegram_message` (как в ТЗ), без переноса кода из ТЗ.

## 5. Обработка входящих параметров (start_param, initData)

### 5.1 Парсинг initData на фронте

- **ТЗ:** `parseInitData()` — user, start_param, auth_date, hash, query_id из `initDataUnsafe`.
- **У вас:** в auth.js сохраняются `user.id` и `language_code`; в api.js для запросов берётся сырая строка `initData` и отправляется в заголовке `X-Telegram-Init-Data`.

**Реализовано:** данные пользователя и initData у вас уже используются. **Взять из ТЗ можно:** чтение `start_param` (например для реферальных ссылок `t.me/bot?start=ref_123`) — одна функция на фронте, которая возвращает `Telegram.WebApp.initDataUnsafe.start_param`, и при необходимости передача его на бэкенд.

**Сделано:** в api.js добавлена `getTelegramStartParam()` (возвращает строку или '' при отсутствии — без краша). Заголовок `X-Telegram-Start-Param` передаётся во все вызовы `api()`. В `create_booking` опционально принимается заголовок и пишется в `log_action(details["start_param"])` для аналитики; при отсутствии заголовка логика без изменений.

### 5.2 Валидация initData на сервере

- **ТЗ:** Node.js, crypto, HMAC-SHA256 по алгоритму Telegram.
- **У вас:** **реализовано в Python** — `services/telegram_auth.py`: `validate_init_data()`, `get_user_id_from_init_data()`, тот же алгоритм (data-check-string, secret_key из WebAppData + bot_token, compare_digest).

**Вывод:** у вас валидация уже есть и сделана под ваш стек. **Код из ТЗ не переносить** — у вас корректная и безопасная реализация.

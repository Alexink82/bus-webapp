# План оптимизации — согласованная версия

**Цель:** LCP < 800 мс, повторные открытия < 200 мс при сохранении всего визуального богатства (blur bottom-sheet, анимированные бейджи, чекмарк с draw, haptic, segmented control, role-shell, графики, QR и т.д.).

**Рекомендуемый порядок:** День 1–2 → A1 (Vite), День 3–4 → A2+A3, День 5–7 → B4+B5, дальше C и D.

## Уточнения и корректировки (чтобы не ломать текущее поведение)

### 1. `base` в vite.config.js

- Сейчас в плане: `base: '/webapp/'`.
- Нужно проверить, как отдаётся фронт в проде: если приложение открывается по корню (например `https://your-app.onrender.com/`), то `base` должен быть `'/'`.
- Если статика всегда по пути `/webapp/` (например `https://your-app.onrender.com/webapp/index.html`), то `base: '/webapp/'` верный.
- **Рекомендация:** перед A1 зафиксировать текущий URL фронта и задать `base` под него; при деплое на Render проверить, что `app.mount("/webapp", ...)` или `mount("/", ...)` соответствует этому base.

### 2. Роль-shell и entry-файлы

- В плане в profile-entry и admin-entry фигурирует `role-shell.js` как новый компонент.
- Сейчас роль-шелл (вкладки Бронь | Профиль | Диспетчер | Админ) реализован в **nav.js** (заполнение `#roleShellPlaceholder`, показ по ролям).
- **Рекомендация:** не заводить отдельный `role-shell.js`, а оставить в entry импорт **nav.js**. Если позже захочется вынести роль-шелл в отдельный модуль — сделать это рефакторингом после того, как Vite уже работает.

### 3. cache.js и заголовки API (Telegram)

- Сейчас запросы к API идут через обёртку с заголовками `X-Telegram-User-Id`, `X-Telegram-Init-Data` (в api.js или аналоге).
- В плане A2 `cachedFetch` показан через голый `fetch(url)`. Так кэш будет работать, но запросы без этих заголовок могут вернуть 401/403.
- **Рекомендация:** в A2 реализовать кэш так: **внутри существующей обёртки api()** (или там, где уже добавляются заголовки) — кэш по ключу `url + опционально initData` (или просто url, если сессия одна). То есть не заменять api() на сырой fetch в cache.js, а добавить слой кэша внутри api.js или обернуть вызовы api() в getWithCache(api, url, ttl). Тогда и кэш, и заголовки сохраняются.

### 4. QR-код: CDN vs npm

- Сейчас используется **qrcode.js** с CDN (глобальный `QRCode`, вызов `new QRCode(el, { text, width, height })`).
- В плане B5 предлагается `await import('qrcode')` — это npm-пакет **qrcode**, у него другой API (toCanvas, toDataURL), а не `new QRCode(el, options)`.
- **Рекомендация:** два варианта:
  - **Вариант A:** оставить библиотеку с CDN и подгружать её динамически через создание `<script src="...">` при первом открытии модалки с QR (lazy load без смены API).
  - **Вариант B:** поставить npm `qrcode`, в модалке при открытии делать `const QRCode = (await import('qrcode')).default` и генерировать QR через `QRCode.toCanvas(canvas, url, { width, height })` (или toDataURL и вставить в img), заменив текущий вызов `new QRCode(el, ...)`.
- Оба варианта дают lazy load; B — единообразие с остальными npm-зависимостями после Vite.

### 5. Эндпоинт dashboard — путь и имя роутера

- В плане указан пример в `user_router.py` и путь `/api/dashboard`.
- У нас роутер пользователя — **api/user.py** с префиксом **/api/user**. Логично сделать эндпоинт **GET /api/user/dashboard** в этом же файле, чтобы не плодить лишние роутеры и сохранить авторизацию через тот же `Depends(get_verified_telegram_user_id)`.
- **Рекомендация:** в A3 добавить в **backend/api/user.py** эндпоинт `@router.get("/dashboard")`; полный путь будет `/api/user/dashboard`. На фронте вызывать `cachedFetch(base + '/api/user/dashboard')` (с учётом BASE_URL, если он есть).

### 6. Инвалидация кэша при действиях

- В A2 в чек-листе указана инвалидация после отмены/переноса/архивации.
- Стоит явно добавить инвалидацию также:
  - после создания заявки (переход на success) — сбросить `/api/user/bookings` и `/api/user/dashboard`;
  - после добавления/редактирования/удаления пассажира — сбросить `/api/user/passengers` и `/api/user/dashboard`;
  - после смены статуса в админке (массовая отмена и т.д.) — сбросить кэш списка заявок админки, если он будет кэшироваться.
- Так повторные открытия профиля/главной всегда получат актуальные данные после любого действия.

### 7. sw.js и путь к статике после Vite

- В B6 пример: `event.request.url.includes('/webapp/') || event.request.url.includes('/dist/')`.
- После Vite статика будет отдаваться с одного пути (например `/webapp/` при base `/webapp/`), а не из двух. Имена файлов будут с хешем (например `profile-entry-a1b2c3.js`).
- **Рекомендация:** в sw.js при Cache First проверять путь статики так же, как фактически раздаёт бэкенд (например `request.url.includes('/webapp/')` или то, что в `base`). Не привязываться к `/dist/` в URL, если в проде раздаётся уже из dist, но по пути `/webapp/`.

## Итоговый чек-лист (с учётом уточнений)

- **A1** — Vite настроен: root webapp, multi-page input, 7 entry-файлов, в HTML только один `<script type="module">`, base согласован с деплоем, build и preview проверены.
- **A2** — cache.js интегрирован **в обёртку api()** (или эквивалент), чтобы сохранить Telegram-заголовки; инвалидация при создании/отмене заявки, изменении пассажиров, действиях в админке.
- **A3** — эндпоинт **GET /api/user/dashboard** в api/user.py; фронт переведён на один запрос к dashboard вместо водопада.
- **B4** — переключатель режима в профиле; performance.js с авто-определением и учётом prefers-reduced-motion; в экономном режиме отключены blur/тяжёлые анимации/автозагрузка Chart.
- **B5** — Chart.js загружается только при открытии вкладки «Статистика»; QR — только при открытии модалки (CDN скрипт или npm qrcode с адаптацией API).
- **B6** — sw.js обновлён под Cache First для статики по фактическому пути (например /webapp/); PWA проверена после build.
- **C** — Preload для profile и booking; Redis-кэш для dashboard (и при необходимости для списков админки); ETag + 304; пагинация «Все заявки» (limit 50 + «Загрузить ещё»).
- **D** — по мере необходимости: виртуализация списков, Web Vitals, A/B, TypeScript.

План в таком виде считаю согласованным и готовым к реализации по шагам A1 → A2+A3 → B4+B5 → B6 → C → D.

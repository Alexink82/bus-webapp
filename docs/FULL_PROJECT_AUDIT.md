## 1. Пользователь (пассажир)

**Сценарий:** зайти из Telegram или браузера → выбрать маршрут и дату → забронировать → увидеть заявку и при необходимости отменить.

- **Главная:** маршруты из API, города в селектах, обмен «Откуда/Куда», блок «Актуальная информация» (погода, граница) — заполняется при старте и по расписанию; при пустом кэше отдаются заглушки. Логотип, переключатель темы, одна кнопка закрытия (в Telegram скрыта).
- **Бронирование:** два шага (пассажиры → контакт и оплата), валидация ФИО и даты рождения, кнопка «Забронировать» блокируется после клика. Редирект на success с booking_id.
- **Успех:** номер заявки, отмена заявки (только владелец), ссылки «Новый поиск» и «Мои заявки».
- **Профиль:** список заявок и сохранённых пассажиров; добавление/удаление пассажиров; отмена заявки из списка. Без Telegram показывается приглашение войти.
- **FAQ:** поиск по вопросам, ссылка в поддержку (Telegram).
- **Уведомления:** при заданном WEBAPP_URL в сообщениях бота о создании/смене статуса заявки есть кнопка «Открыть заявки» (web_app).

**Проверить вручную:** чек-лист раздел 1 и 6 в `docs/testing/MANUAL_TEST_CHECKLIST.md` (главная, бронирование, успех, профиль, FAQ, уведомления и start_param).

## 2. Администратор

**Сценарий:** войти под учётной записью из ADMIN_IDS или bot_roles → смотреть статистику и логи, экспортировать данные.

- **Доступ:** admin.html; при отсутствии user_id — сообщение «Войдите через Telegram»; при user_id не из списка админов — «Нет доступа (не админ)». Роли: ADMIN_IDS (env) + bot_roles (is_admin).
- **Статистика:** период, количество заявок, сумма (BYN).
- **Экспорт CSV:** кнопка «Экспорт CSV», скачивание файла по выбранному периоду.
- **Логи:** последние события (время, уровень, источник, action). При создании заявки с start_param он попадает в details.
- **Логотип:** выведен над шапкой админки (design-system).

**Что не реализовано в UI:** управление диспетчерами (добавление/удаление) — API есть (GET/POST /api/admin/dispatchers), интерфейса в админке нет. По ROADMAP — «UI для управления диспетчерами» в бэклог.

**Проверить:** раздел 3 чек-листа (доступ, статистика, экспорт, логи).

## 3. Диспетчер

**Сценарий:** войти под учётной записью из dispatchers или bot_roles → видеть новые заявки, брать в работу, менять статус, смотреть статистику.

- **Доступ:** dispatcher.html; при отсутствии user_id — «Войдите через Telegram»; при user_id не диспетчера — «Нет доступа (вы не диспетчер)». Роли: таблица dispatchers (is_active) + bot_roles (is_dispatcher).
- **Вкладки:** Новые, В работе, Поиск, Статистика. В «Новые» — кнопка «Взять в работу»; во «В работе» — смена статуса (Оплачено, Билет отправлен, Завершено). Поиск по номеру заявки или фамилии.
- **WebSocket:** при создании новой заявки подключённые диспетчеры получают уведомление без перезагрузки. При BOT_TOKEN требуется первый message с type=auth и init_data; dispatcher_id сверяется с user_id из initData.
- **Логотип и тема:** на странице диспетчера есть логотип и переключатель светлой/тёмной темы.

**Проверить:** раздел 2 чек-листа (вход, новые заявки, конфликт «взял другой», в работе, статистика, поиск, WebSocket).

## 4. Программист

**Архитектура:** FastAPI (backend), Vanilla JS + HTML/CSS (frontend), PostgreSQL, WebSocket для диспетчеров. Один процесс, статика раздаётся с того же сервера.

- **Исправления из PRODUCTION_READINESS и AUDIT:** проверка initData, ограничение GET /api/bookings/{id}, отмена по правам, blacklist при создании брони, rate limit, проверка auth в WebSocket, повтор при IntegrityError по booking_id, валидация дат и коды ошибок (invalid_date_format, invalid_birth_date и др.), escapeHtml при вставке в innerHTML, маппинг ERROR_MESSAGES на фронте, отключение кнопки «Забронировать» при отправке.
- **Тесты:** backend/tests (pytest), conftest, test_api — базовые проверки API. Ручная проверка по MANUAL_TEST_CHECKLIST.
- **Документация:** README, PRODUCTION_READINESS, ROADMAP, UI_IMPROVEMENTS_PLAN, THEME_SWITCHER, COMPARISON_TZ_DUCK_VS_BUS, AUDIT_MULTI_ANGLE, DEPLOY_GITHUB_RENDER, тестовый чек-лист.
- **Долги:** CORS allow_origins=["*"]; двойной commit в get_db и в части хендлеров (не ломает); README описывает деплой без Docker — в реальности используется Dockerfile в корне.

**Рекомендации:** актуализировать README под текущий деплой (Docker, Render, репозиторий bus-webapp); при желании сузить CORS; добавить в бэкенд timeout для внешних вызовов (OpenWeather, border) и обрабатывать таймауты в /api/news.

## 5. Дизайнер

**Единообразие:** главная, профиль, успех, FAQ, бронирование, диспетчер — общая шапка (логотип СТОЛИЦА, заголовок страницы, переключатель темы, кнопка закрытия где нужна), общий нижний бар (Бронь, Профиль, при наличии ролей — Диспетчер, Админ, FAQ). Стили: main.css + telegram-theme.css; админка — design-system.css, логотип общий.

**Тема:** светлая/тёмная, переключатель в шапке, сохранение в localStorage, при первом визите — prefers-color-scheme. В светлой теме логотип инвертируется (filter: invert(1)). Плавные переходы при смене темы.

**Отступы и наложение:** шапка с margin-bottom; кнопки закрытия и темы с фиксированным размером и gap; нижнее меню с safe-area-inset-bottom; шрифты в меню адаптивные (clamp), увеличены для читаемости.

**Рекомендации:** проверить контраст текста по WCAG (например 4.5:1 для основного текста); при желании добавить опцию «Как в системе» для темы; meta theme-color уже обновляется в theme.js при смене темы.

## 7. Владелец продукта (Product owner)

**Реализовано по сравнению с задуманным:** бронирование за несколько минут, сохранённые пассажиры, панель диспетчера в реальном времени, уведомления в Telegram с кнопкой в Web App, PWA-остов (manifest, sw), оплата — заглушка для тестов. Роли (админ, диспетчер) и разграничение доступа работают.

**Бэклог (ROADMAP и др.):** CORS сузить под прод; README обновить; idempotency при бронировании (опционально); реальный WebPay с проверкой подписи; UI админки для управления диспетчерами и при желании архивация; кэш маршрутов при переходе на API; meta description и OG при публичном доступе по URL.

**Метрики:** start_param пишется в логи при создании заявки — можно анализировать переходы по реферальным ссылкам.

## 8. DevOps

**Деплой:** репозиторий GitHub (bus-webapp), Render — Web Service из Docker (Dockerfile в корне), автодеплой при push в main. Переменные окружения задаются в Render (BOT_TOKEN, DATABASE_URL, OPENWEATHER_API_KEY, WEBAPP_URL и др.). PostgreSQL — отдельный сервис Render или внешний; при первом запуске выполняется init_db и при необходимости seed_db (вручную или через команду).

**Наблюдаемость:** логирование запросов и ошибок в main.py; логи приложения в Render; health: GET /api/health. Специального мониторинга (метрики, алерты) в коде нет.

**Рекомендации:** в README указать актуальные шаги деплоя (Docker, Render, репо bus-webapp); при необходимости добавить лёгкую проверку БД в /api/health; настроить алерты в Render при падении сервиса.

## 9. Безопасность

**Сделано:** проверка Telegram initData при заданном BOT_TOKEN; GET /api/bookings/{id} без прав отдаёт только публичные поля; отмена заявки только владелец/диспетчер/админ; blacklist по phone и user_id при создании брони; rate limit по IP; WebSocket — auth по init_data и проверка dispatcher_id; проверка подписи/секрета для payment callback (WEBPAY_CALLBACK_SECRET); экранирование при выводе в HTML (escapeHtml).

**Рекомендации:** сузить CORS под прод (по желанию); рассмотреть заголовок Content-Security-Policy; при появлении пользовательского контента в БД сохранять практику экранирования и валидации на бэкенде.

---

## 10. Доступность (a11y)

**Сделано:** кнопки закрытия и переключатель темы с aria-label и title; семантика (header, main, nav); контраст текста на фоне в обеих темах.

**Рекомендации:** проверить контраст по WCAG (например WebAIM Contrast Checker); при необходимости добавить aria-live для динамических сообщений и aria-hidden для скрытых блоков; убедиться, что все интерактивные элементы доступны с клавиатуры.

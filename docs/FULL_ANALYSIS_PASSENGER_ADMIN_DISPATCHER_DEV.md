# Полный анализ проекта bus-booking-webapp

**Дата:** 2025-03-10  
**Подход:** Анализ с позиций пассажира, админа, диспетчера, разработчика и смежных специалистов. Только факты по коду и поведению системы.

---

## 1. Взгляд пассажира

### 1.1 Что работает
- Поиск рейсов по маршруту и дате, выбор рейса, переход в бронирование.
- Форма бронирования: пассажиры, телефон, способ оплаты, опция «для другого человека», сохранение в профиль.
- Валидация: дата рождения (DD.MM.YYYY, месяц 1–12), паспорт (МР/НВ и т.д.), телефон РБ/РФ с префиксом и пошаговой подсветкой ошибок.
- ФИО автоматически переводятся в верхний регистр.
- После создания заявки — страница успеха с номером заявки, отмена (если >2 ч до отправления), ссылки «Мои заявки» и «Новый поиск».
- В профиле: список заявок, сохранённые пассажиры, настройки (язык RU/EN/BE, тема, «уменьшить анимации»).
- Модальное окно «Подробнее» по заявке с полными данными и кнопкой «Перенести дату» (переход на главную с подставленными городами).
- При первом заходе показывается окно согласия на обработку данных (один раз, флаг в localStorage).
- Язык: определение по Telegram/системе, смена в настройках; многие экраны переведены (главная, успех, FAQ, профиль частично).

### 1.2 Ошибки и неточности
- **Ошибки только на русском.** Все сообщения в `api.js` (ERROR_MESSAGES) и текст «Временная ошибка сервера» — русские. При выборе EN/BE пользователь всё равно видит русские ошибки.
- **Часть текстов не переведена.** В форме бронирования (booking.html / booking.js): «Шаг 1. Пассажиры», «Количество пассажиров», «Бронь для другого человека», подсказки, «Продолжить», «Шаг 2. Контакт и оплата», «Итого: … BYN», все сообщения валидации («Заполните фамилию…», «Неверная дата…» и т.д.) — захардкожены по-русски.
- **Модальное окно «Подробнее»:** заголовки полей («Маршрут:», «Дата и время:», «Пассажиры:», «Контактный телефон:», «Стоимость:», «Статус:») и кнопка «Страница заявки» не используют `t()`.
- **FAQ при lang=be:** бэкенд отдаёт только `question_ru` / `question_en`; для `be` фронт в faq.html запрашивает `question_be`/`answer_be`, которых в API нет — в ответе приходит только `question`/`answer` (бэкенд подставляет en при lang≠ru). То есть белорусский FAQ по сути не поддерживается, показывается английский.
- **Главная страница:** после загрузки маршрутов подстановка `from`/`to` из URL работает только если города есть в списке остановок; при несовпадении (опечатка, другой регистр) поля остаются пустыми без сообщения.
- **Расписание:** подпись «Минск (UTC+3)» и текст про рейсы — только русский.
- **Блок «Актуальная информация»:** заголовок и контент (border, weather) — без i18n.

### 1.3 UX и доступность
- Нет явной индикации загрузки при отправке бронирования (кнопка блокируется, но спиннера/текста нет).
- Нет единого обработчика «нет сети» с понятным сообщением.
- Часть кнопок/полей есть в разметке без `aria-label` / подписей под текущий язык.

## 2. Взгляд диспетчера

### 2.1 Что работает
- Вход по Telegram; без входа показывается предупреждение.
- Вкладки: «Новые», «В работе», «Поиск», «Статистика».
- Список новых заявок, кнопка «Взять в работу».
- По заявкам «В работе»: смена статуса на «Оплачено», «Билет отправлен», «Завершено».
- Поиск по номеру заявки (GET /api/bookings/{id}); ответ выводится в карточке.
- Статистика за сегодня по своим заявкам.
- WebSocket: при новой заявке обновляется список и даётся тактильный отклик.

### 2.2 Критические несоответствия
- **Отмена заявки диспетчером и причина отмены.**  
  В бэкенде:
  - `POST /api/bookings/{id}/cancel` — для диспетчера обязательна причина (`reason`), она пишется в `cancel_reason`.
  - `POST /api/dispatcher/bookings/{id}/status` с `status: "cancelled"` — отменяет заявку, но **не принимает и не сохраняет причину** и не заполняет `cancel_reason`.  
  В интерфейсе диспетчера **нет кнопки «Отменить»** и нет вызова `/api/bookings/.../cancel`. Есть только смена статуса на «Оплачено» / «Билет отправлен» / «Завершено».  
  **Итог:** правило «причина отмены от диспетчера» реализовано только в одном из двух способов отмены; тем способом, которым пользуется интерфейс диспетчера, причина не указывается и не сохраняется.
- **Диспетчер не может отменить заявку через UI.** Кнопки отмены в панели диспетчера нет; отмена возможна только через смену статуса на `cancelled`, при этом без причины.

### 2.3 Прочие замечания
- Весь интерфейс диспетчера на русском: заголовки, вкладки, кнопки, сообщения («Нет новых заявок», «Взять в работу», «Оплачено» и т.д.). i18n не используется.
- Поиск по заявке: запрос к `/api/bookings/{id}` идёт с теми же `headers` (X-Telegram-User-Id, Init-Data); эндпоинт при наличии прав отдаёт полные данные. Если убрать заголовки — запрос анонимный, но тогда вернётся урезанный ответ (без пассажиров и телефона). Фактически поиск выполняется от имени диспетчера — ок.
- При ошибках показывается `alert(e.message)`; коды ошибок не маппятся в понятные сообщения (как в api.js для пассажира).

## 3. Взгляд администратора

### 3.1 Что работает
- Вход по Telegram; проверка прав (ADMIN_IDS + bot_roles.is_admin).
- Статистика, логи, список администраторов, список диспетчеров.
- Добавление админа по telegram_id, добавление/деактивация диспетчера.
- Экспорт заявок в CSV.
- Архивация старых заявок (POST /archive).

### 3.2 Замечания
- Весь UI на русском, без i18n.
- Ошибки («Нет доступа (не админ).», «Ошибка экспорта» и т.д.) — захардкожены.
- Нет возможности отменить заявку от имени системы с указанием причины через админку; отмена только через те же API, что и у диспетчера/пассажира.

## 4. Взгляд разработчика (код и архитектура)

### 4.1 Бэкенд
- **Двойной commit.** В `database.get_db()` после `yield` выполняется `await session.commit()`. Во многих обработчиках (booking create/cancel, dispatcher take/status и др.) перед return делается свой `await db.commit()`. В итоге один и тот же запрос даёт два commit. Второй обычно пустой, но это лишний вызов и размытая ответственность за транзакцию; при доработках легко ошибиться (например, читать после своего commit без повторного запроса).
- **Часовой пояс при отмене.** В `cancel_booking` используется `get_local_time()` (Minsk, UTC+3) и наивный `datetime` по `b.date` и `b.departure`. Сравнение корректно только если сервер и все даты трактуются в одной и той же зоне (Minsk). При переносе сервера или смене логики дат возможны сдвиги.
- **Время отправления в БД.** Поле `departure` — строка (например "16:30"); парсинг через `split(":")` без проверки длины/формата при неверных данных может дать исключение (частично прикрыто try/except, но не по всему пути).
- **Валидация телефона на бэкенде.** В `validators.validate_phone` — только длина цифр 9–15. Нет проверки префикса +375/+7 и длины номера по странам; фронт строже. Возможен приём «легитимных» по длине, но бессмысленных номеров.
- **FAQ: только ru/en.** Модель `FAQItem`: `question_ru`, `question_en`, `answer_ru`, `answer_en`. Параметр `lang` в API обрабатывается как «ru иначе en»; белорусского нет.
- **Миграции.** Нет Alembic/миграций; изменения схемы — ручные ALTER в `database.init_db()` и повторный запуск. Риск расхождений между средами и версиями.

### 4.2 Фронт
- **Дублирование API-вызовов.** В profile.js используется свой `apiFn` (fetch с теми же заголовками), а не общий `api()` из api.js — дублирование логики и риска расхождений при смене заголовков/обработки ошибок.
- **showBookingDetailsModal и showAppModal.** Модальное окно «Подробнее» формирует HTML со ссылками «Перенести дату» и «Страница заявки». При клике по ссылке происходит переход, страница выгружается — отдельно закрывать модалку не обязательно. Но контент модалки (в т.ч. данные заявки) собирается через конкатенацию строк; при появлении в API неэкранированных символов возможен XSS. Сейчас в ответе используются `esc()` для полей — нужно не забывать экранировать все новые поля.
- **formatPhoneInput (input-masks.js).** При вводе, например, «9» после «+7 » логика может вести себя неочевидно (разные ветки для 375 и 7). Стоит проверить граничные случаи (только +, только 8, 80…).
- **Индекс (главная).** Подстановка from/to из URL выполняется в then после fetch маршрутов; если fetch упадёт, подстановка не выполнится (нет в catch).

### 4.3 Безопасность
- **CORS:** `allow_origins=["*"]` — в проде лучше ограничить домен.
- **Раскрытие данных по booking_id.** GET /api/bookings/{id} без авторизации отдаёт часть данных (маршрут, дата, время, кол-во пассажиров, сумма, статус). Номер заявки предсказуем по формату (BK-ddmmyy-HHMMSS). Возможен перебор и сбор информации о заявках.
- **XSS:** В рассмотренных местах вывод в DOM идёт через `esc`/escapeHtml. Контент согласия (consent.js) — фиксированные строки по языку, не от API. Новый контент из API/пользователя нужно по-прежнему экранировать.
- **SQL:** Параметризованные запросы/ORM; явных рисков SQL-инъекций по коду не видно.

## 5. Другие специалисты

### 5.1 Тестировщик
- Нет автоматических E2E; только unit/интеграционные тесты в `backend/tests` (здоровье, маршруты, валидация, отмена без авторизации и т.д.).
- Сценарии «пассажир — бронирование — отмена», «диспетчер берёт заявку и меняет статус», «админ добавляет диспетчера» не покрыты автотестами.
- Ручные сценарии описаны в docs/testing (MANUAL_TEST_CHECKLIST.md и др.), но не факт, что все граничные случаи (неверный телефон, дата, отмена за <2 ч) регулярно прогоняются.

### 5.2 Аналитик / продукт
- Бизнес-правило «отмена только через диспетчера с указанием причины» в продукте не выполнено: в UI диспетчера отмена без причины и без сохранения причины.
- Заявленную поддержку трёх языков (RU/EN/BE) FAQ и часть экранов не поддерживают полностью (ошибки, диспетчер, админка, многие строки бронирования).

### 5.3 DevOps / эксплуатация
- Rate limit хранится в памяти (словарь по IP); при рестарте сбрасывается; при нескольких инстансах — свой лимит на каждый инстанс.
- Нет явного health-check на зависимость от БД (например, простой SELECT в /api/health).
- Логирование запросов и ошибок есть; структурированных логов (request_id, user_id) для трассировки нет.

## 7. Рекомендации (по приоритету)

1. **Диспетчер и отмена:**  
   Либо добавить в UI диспетчера кнопку «Отменить» с полем «Причина» и вызовом `POST /api/bookings/{id}/cancel` с `body: { reason }`, либо в `POST /api/dispatcher/bookings/{id}/status` при `status === "cancelled"` принимать и сохранять `reason` в `Booking.cancel_reason`. Привести одно правило к одному способу отмены.

2. **Ошибки и языки:**  
   Вынести тексты ошибок в i18n (или общий словарь по коду + маппинг на текущий язык) и использовать их в api.js и везде, где показываются сообщения пользователю.

3. **FAQ для BE:**  
   Либо добавить в модель и API question_be/answer_be и отдавать их при lang=be, либо явно документировать, что для be показывается en (и не обещать полную поддержку BE в FAQ).

4. **Commit:**  
   Либо убрать commit из get_db и оставить его только в обработчиках, либо убрать явные commit из обработчиков и полагаться на get_db. Зафиксировать единое правило в коде и в документации.

5. **CORS и раскрытие заявок:**  
   В проде ограничить CORS списком доменов. Рассмотреть обязательную авторизацию для GET /api/bookings/{id} или капчу/лимит при неизвестном пользователе.

6. **Миграции:**  
   Ввести миграции (например, Alembic) и переносить все изменения схемы через миграции, а не через разовые ALTER в init_db.

Документ можно использовать как чек-лист для исправлений и для приведения продукта в соответствие с заявленными правилами и поддержкой языков.

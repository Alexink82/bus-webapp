# Vite: сборка и проверки

## Текущее состояние

- Все 7 страниц переведены на entry-модули: в каждом HTML подключён один `script type="module" src="./js/xxx-entry.js"`, inline-логика вынесена в отдельные файлы (например `profile-init.js`, `admin.js`). Vite собирает бандлы в `dist/assets/*.js`.
- **Клиентский кэш API (A2):** в `api.js` — in-memory кэш GET-запросов (TTL 60 с), автоматическая инвалидация при мутациях (брони, пассажиры, профиль). Опция `skipCache: true` и `window.apiInvalidateCache(prefix)` для ручной очистки.
- **Эндпоинт dashboard (A3):** `GET /api/user/dashboard` возвращает `profile`, `passengers`, `bookings` одним запросом; страница профиля использует его для первой загрузки и обновления.
- **B4 — Режим интерфейса:** В настройках профиля переключатель «Нормальный» / «Экономный» (localStorage `uiMode`). Авто-режим при первом запуске по `hardwareConcurrency`/`deviceMemory`. Хелпер `isEconomyMode()` (учитывает также «Уменьшить анимации» и `prefers-reduced-motion`). В экономном режиме: без blur у модалок, статичный скелетон, укороченная/выключенная анимация чекмарка на success, QR в профиле только по кнопке «Показать QR», префетч отключён.
- **B5 — Lazy-load:** Библиотека qrcode на странице профиля подгружается только при открытии модалки с QR (или по кнопке «Показать QR» в экономном режиме). IMask по-прежнему из CDN в HTML (при желании можно вынести в lazy).
- **B6 — Service Worker:** `webapp/public/sw.js` — стратегия Cache First для запросов того же origin, кроме `/api/`. Версионирование кэша (`bus-booking-v2`), при активации нового SW старые кэши удаляются. Регистрация в `api.js` при загрузке (HTTPS или localhost).

## Команды

| Команда | Действие |
|--------|----------|
| `npm run build` | Сборка Vite → каталог `dist/` |
| `npm run test:build` | Сборка + проверка наличия страниц и `assets/*.js` |
| `npm run dev` | Dev-сервер Vite (порт 5173), прокси `/api` и `/ws` на бэкенд |
| `npm run copy-dist` | Копирование `dist/` в `webapp/` (перезаписывает HTML/CSS в webapp; использовать только при деплое, когда статика отдаётся из webapp) |

## Рекомендуемый деплой

- **Вариант A:** отдавать статику из **`dist/`** (корень статики = `dist/`), не копировать в webapp. Тогда после каждой сборки в `dist/` лежат готовые `index.html`, `dispatcher.html` и т.д. с путями к хешированным файлам в `assets/`.
- **Вариант B:** при сборке в Docker выполнять `npm run build` и затем копировать только **`dist/assets/`** в `webapp/assets/`, а HTML не перезаписывать (чтобы не потерять ссылки на entry в исходниках). Тогда бэкенд по-прежнему отдаёт из `webapp/`, а бандл диспетчера будет по пути `webapp/assets/dispatcher-xxx.js` — для этого в `dispatcher.html` в проде должен быть подставлен путь к бандлу (например, через шаблон или отдельный шаг подстановки при деплое).

## Проверка взаимосвязей

1. **Сборка:** `npm run test:build` — должен завершиться без ошибок, в `dist/` должны быть все 7 HTML и каталог `assets/` с `.js`/`.css`.
2. После сборки открыть любую страницу из `dist/` (или через `npm run preview`) — скрипты подгружаются как бандлы из `dist/assets/*.js`.

## C и D (реализовано)

- **C.1 Preload:** На главной (index) при загрузке в неэкономном режиме добавляются prefetch для profile.html и booking.html (в index-init.js).
- **C.2 Redis-кэш dashboard:** Ответ GET /api/user/dashboard кэшируется в Redis по ключу `dashboard:{user_id}` (TTL 60 с). Инвалидация при изменении профиля, пассажиров, создании/отмене брони и при админской массовой отмене (`services/dashboard_cache.py`, вызовы из `api/user.py`, `api/booking.py`, `api/admin.py`).
- **C.3 ETag + 304:** GET /api/user/dashboard отдаёт заголовок ETag; при совпадении If-None-Match возвращается 304. Заголовок Cache-Control: private, max-age=0.
- **D Виртуализация:** В профиле списки заявок и пассажиров при числе элементов > 10 показывают первые 10 и кнопку «Показать ещё (N)» (PROFILE_LIST_PAGE_SIZE в profile.js).
- **D Web Vitals:** Подключён web-vitals.js (LCP, FID, CLS через PerformanceObserver). Логи в console; при заданном `window.__WEB_VITALS_BEACON_URL` — отправка через sendBeacon (index-entry.js).
- **D A/B экономного режима:** При первом визите без сохранённого uiMode сохраняется когорта A/B (localStorage economyAbCohort). Когорта B по умолчанию получает экономный режим; A — по устройству. Доступен `window.getEconomyCohort()` для опциональной аналитики.

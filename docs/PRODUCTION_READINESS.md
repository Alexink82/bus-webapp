# Анализ готовности к продакшену (Bus Booking Web App)

Документ: глубокий анализ проекта, краш-тесты, безопасность, неясные вопросы и рекомендации с учётом деплоя на Render (free tier).

---

## 1. Контекст и допущения

- **Сервер**: Render.com (free tier) — один инстанс, холодный старт после ~15 мин без трафика, ограничения по памяти/CPU.
- **БД**: PostgreSQL (общая с bus-bot): таблицы `bookings`, `bot_roles`; веб-приложение дополнительно использует `user_profiles`, `saved_passengers`, `dispatchers`, `log_entries`, `faq_items`, `cached_data`, `webpay_transactions`.
- **Идентификация**: при заданном `BOT_TOKEN` требуется заголовок `X-Telegram-Init-Data` и криптографическая проверка подписи на бэкенде (`services/telegram_auth`, `api/auth_deps`). Без BOT_TOKEN (разработка) достаточно `X-Telegram-User-Id`.
- **Роли**: админ — `ADMIN_IDS` (env) + таблица `bot_roles` (is_admin); диспетчер — таблица `dispatchers` (is_active) + `bot_roles` (is_dispatcher). При старте вызывается `load_roles(db)`.

## 2. Критические проблемы

### 2.1 Безопасность

| Проблема | Где | Риск | Рекомендация |
|----------|-----|------|--------------|
| **Нет проверки initData** | Весь API | Любой может подставить произвольный `X-Telegram-User-Id` и действовать от имени пользователя/диспетчера/админа. | Реализовать проверку подписи Telegram (BOT_TOKEN + initData → hash) на бэкенде для запросов с заголовком; при несовпадении возвращать 401. |
| **GET /api/bookings/{booking_id} без авторизации** | api/booking.py | Любой, кто знает или подбирает номер заявки (BK-100324-120000), видит полные данные: ФИО, паспорт, телефон, маршрут. | Либо требовать заголовок и отдавать детали только владельцу (contact_tg_id) или диспетчеру/админу; либо отдавать только публичные поля (номер, статус, маршрут) без персональных данных. |
| **POST /api/bookings/{booking_id}/cancel без авторизации** | api/booking.py | Любой может отменить любую заявку по номеру. | Разрешить отмену только: владельцу (contact_tg_id) по заголовку или диспетчеру/админу. |
| **POST /api/payment/callback без подписи** | api/payment.py | В проде WebPay присылает callback с подписью. Сейчас mock принимает любой POST — злоумышленник может пометить заявку как оплаченную. | Для продакшена: проверять подпись WebPay; для mock — опционально проверять секретный ключ из env. |
| **WebSocket /ws/dispatcher/{dispatcher_id}** | api/websocket.py | В path передаётся `dispatcher_id`; сервер не проверяет, что клиент действительно этот диспетчер (можно подставить чужой ID и получать чужие уведомления). | При подключении требовать заголовок или первый message с токеном/initData и сверять dispatcher_id с проверенным user_id. |

### 2.2 Целостность данных и краш-сценарии

| Сценарий | Где | Что происходит | Защита |
|----------|-----|----------------|--------|
| **Неверный формат даты в бронировании** | create_booking: `date.fromisoformat(body.departure_date)` | При строке не "YYYY-MM-DD" — ValueError → 500, в логах traceback. | Обернуть в try/except, возвращать 400 с кодом `invalid_date_format`. |
| **Неверный формат даты рождения в пассажире** | validate_passenger / calculate_booking_totals | В validators уже есть try/except для birth_date; в price_calc — date.fromisoformat(bd) без try — при неверном формате из фронта возможен сбой. | В calculate_booking_totals обрабатывать исключение или гарантировать, что до сюда доходят только уже провалидированные данные. |
| **Дубликат booking_id** | generate_booking_id() | Формат BK-{ddmmyy}-{HHMMSS}. Два запроса в одну секунду → один и тот же id → конфликт при вставке в БД (IntegrityError). | При конфликте primary key — повторить генерацию (например, добавить случайный суффикс) и повторную вставку (1–2 попытки). |
| **Двойная отправка формы бронирования** | Фронт: двойной клик по «Забронировать» | Два параллельных POST /api/bookings → две заявки с разными id (если id уникален по времени). | На фронте: disable кнопки после первого клика; на бэкенде: опционально idempotency key (например, хэш от маршрут+дата+время+телефон+user_id) и отклонение дубликата за N минут. |
| **Диспетчер дважды нажимает «Взять в работу»** | take_booking | Уже есть проверка: если dispatcher_id другой — 409 already_taken; если тот же — повторная установка того же значения, ок. | Защита есть. |
| **Диспетчер меняет статус чужой заявки** | set_booking_status | Проверка: b.dispatcher_id != dispatcher_id → 403 not_your_booking. | Защита есть. |
| **Админ без ADMIN_IDS** | get_admin_id | Если в env не задан ADMIN_IDS, admin_ids_list пустой → любой запрос с любым заголовком получит 403 not_admin. | При первом деплое обязательно задать ADMIN_IDS в Render. |
| **Нет диспетчеров в БД** | require_dispatcher | Все запросы диспетчера получат 403 not_dispatcher. | Выполнить seed_db.py или добавить диспетчеров через POST /api/admin/dispatchers. |
| **user.py: birth_date при добавлении пассажира** | add_passenger, update_passenger | date.fromisoformat(body.birth_date) при неверной строке → ValueError → 500. | try/except + 400 invalid_birth_date. |

### 2.3 Ошибки и наблюдаемость

- **Middleware** в main.py логирует каждый /api/ запрос и все необработанные исключения с traceback — хорошо.
- **log_action** пишет в log_entries и в stdout с контекстом файл:строка — хорошо.
- Во многих эндпоинтах нет явного try/except: при ошибке БД или валидации FastAPI вернёт 500, middleware залогирует. Для «защиты от дурака» лучше возвращать 400 с понятным кодом вместо 500 при некорректном вводе (даты, типы).

## 3. Несоответствия и неясности в коде

1. **Booking.id** в модели — строка (BK-ddmmyy-HHMMSS), в БД — TEXT PRIMARY KEY. В api/dispatcher и api/booking везде используется как строка — согласовано. В api/payment: `Booking.id == body.booking_id` — верно.
2. **get_db** в database.py после yield делает `await session.commit()`. В хендлерах тоже часто вызывается `await db.commit()`. Получается двойной commit при успехе; при исключении get_db делает rollback. Явный commit в хендлере не обязателен при таком дизайне, но и не ломает.
3. **broadcast_new_booking** в websocket.py вызывается из create_booking после коммита — диспетчеры получают real-time уведомление о новой заявке.
4. **services/roles.py** используется: админ через is_admin(uid) (ADMIN_IDS + bot_roles), диспетчер через get_dispatcher_route_ids(db, uid) (dispatchers + bot_roles).
5. **services/cache.py** (RouteCache) не используется: api/routes.py отдаёт маршруты из core.constants.ROUTES. Кэш можно подключить при переходе на API маршрутов.
6. **config.rate_limit**: middleware в main.py подключён — при превышении лимита возвращается 429.
7. **CORS**: allow_origins=["*"] — для продакшена по желанию сузить до домена веб-приложения.

## 4. Вопросы к тебе (для 100% готовности)

Формулировка по принципу: **что будет, если сделать** и **что будет, если не сделать** — к чему это приведёт.

---

### 1. Проверка initData Telegram на бэкенде

**Если сделать (реализовать проверку подписи initData):**
- Запросы к API с поддельным заголовком `X-Telegram-User-Id` будут отклоняться (401). Войти в админку, диспетчерскую или действовать от имени пассажира сможет только тот, кто реально открыл приложение из Telegram с валидными данными.
- Фронту нужно передавать на бэкенд строку `initData` (или хэш), бэкенд проверяет её через BOT_TOKEN. Небольшое усложнение интеграции и чуть больше трафика.

**Если не сделать:**
- Любой человек может подставить в запросе чужой Telegram ID (например, админа или диспетчера) и получить полный доступ к админке/диспетчерской или к «моим заявкам» другого пользователя. Защиты от подделки идентификации нет.

---

### 2. Ограничение данных GET /api/bookings/{id} без авторизации

**Если сделать (уже сделано: без заголовка/без прав — только публичные поля):**
- По ссылке с номером заявки (например, из письма или мессенджера) без входа видны только номер, статус, маршрут, дата, сумма — без ФИО, паспорта и телефона. Утечка персональных данных по одной только ссылке невозможна.
- Чтобы увидеть полные данные (пассажиры, контакт), нужно открыть ссылку из того же Telegram-аккаунта, что и при бронировании, или войти как диспетчер/админ.

**Если не сделать (отдавать всё без проверки):**
- Любой, кто узнал или подобрал номер заявки (BK-…), может получить полные персональные данные (ФИО, паспорт, телефон). Риск при утечке или пересылке ссылки.

---

### 3. Кто может отменять заявку (владелец / диспетчер / админ)

**Если сделать (уже сделано: только владелец, диспетчер или админ по заголовку):**
- Отменить заявку может только пассажир (contact_tg_id), диспетчер или админ. Случайный человек или злоумышленник с одним лишь номером заявки отменить её не может — получит 401/403.

**Если не сделать (разрешать отмену без проверки):**
- По знанию номера заявки любой мог бы отменять любую бронь (вред, троллинг, конкуренты). Сейчас это уже закрыто проверкой прав.

---

### 4. Подпись или секрет для payment callback (WebPay)

**Если сделать (проверять подпись WebPay или секретный ключ из env):**
- Callback «оплата прошла» принимается только от доверенного источника (WebPay с правильной подписью или наш фронт/сервис с известным секретом). Подделать «успешную оплату» по чужой заявке нельзя.

**Если не сделать:**
- При реальном WebPay в проде без проверки подписи злоумышленник может отправить POST на callback с произвольным transaction_id и пометить заявку как оплаченную без реальной оплаты. С mock-оплатой риск ниже, но при переходе на реальный платёжный шлюз — критично.

---

### 5. Учёт bot_roles в веб-приложении (админы/диспетчеры из панели бота)

**Если сделать (объединить ADMIN_IDS + bot_roles для админов; dispatchers + bot_roles для диспетчеров):**
- Админы и диспетчеры, добавленные через панель bus-bot, автоматически получают доступ в веб-админку и веб-диспетчерскую. Один список ролей для бота и веба, не нужно дублировать настройки в env и в БД.

**Если не сделать:**
- Роли задаются только через ADMIN_IDS в env и таблицу dispatchers в веб-приложении. Изменения в панели бота (добавили/убрали админа или диспетчера) в вебе не отразятся — возможна рассинхронизация и путаница, кто где имеет доступ.

---

### 6. Защита от коллизии номера заявки (дубликат id)

**Если сделать (уже сделано: при IntegrityError — повтор с новым id):**
- Два бронирования, созданные в одну секунду, не приводят к падению с 500: при конфликте primary key генерируется новый id (с суффиксом) и вставка повторяется. Пользователь всегда получает ответ с валидным номером заявки.

**Если не сделать:**
- При двух одновременных запросах возможен одинаковый id → ошибка при вставке в БД (IntegrityError) → 500, одна из заявок не сохранится, пользователь увидит ошибку без понятного номера заявки.

---

### 7. Real-time уведомления диспетчеров (WebSocket при новой заявке)

**Если сделать (вызывать broadcast_new_booking из create_booking):**
- Как только пассажир создал заявку, у всех подключённых диспетчеров (или по маршруту) сразу появляется новая карточка без обновления страницы. Быстрее реакция, меньше ручных обновлений.

**Если не сделать:**
- Диспетчер видит новые заявки только после обновления списка (кнопка «Обновить» или переход по табам). Задержка может быть до нескольких минут, если никто не обновлял — приемлемо при небольшом потоке заявок, но при пиковой нагрузке часть заявок будет «в прятках».

---

### 8. Rate limit (лимит запросов с одного IP)

**Если сделать (in-memory лимит, например 60/мин на создание брони, 120/мин на остальное):**
- Один источник (бот, скрипт, злоумышленник) не сможет завалить API тысячами запросов — после лимита будет 429. Защита от DDoS и случайных циклов на фронте; на Render free меньше шансов уйти в троттлинг или падение по памяти.

**Если не сделать:**
- При всплеске трафика или ошибке в коде (бесконечный retry) один клиент может генерировать очень много запросов. На бесплатном инстансе это может привести к замедлению для всех, рестартам или ограничениям со стороны Render.

---

### 9. Проверка чёрного списка (Blacklist) при бронировании

**Если сделать (проверять user_id и/или phone по таблице blacklist перед созданием брони):**
- Заблокированные пользователи или номера телефонов не смогут оформить новую заявку — получиют 403 с понятным сообщением. Админ может блокировать мошенников или нарушителей в одном месте (бот или веб), и это будет работать и в веб-бронировании.

**Если не сделать:**
- Модель Blacklist есть, но ни один эндпоинт её не использует. Добавление в чёрный список не будет влиять на веб-бронирования — заблокированный пользователь сможет продолжать создавать заявки через веб.

---

### 10. Архивация старых заявок

**Если сделать (добавить is_archived или отдельную таблицу/архив и эндпоинт архивации):**
- Старые заявки можно переносить в архив, основной список и отчёты остаются быстрыми, меньше данных в рабочих выборках. Админ может запускать архивацию по кнопке или по расписанию.

**Если не сделать:**
- Все заявки навсегда остаются в одной таблице. Со временем растёт объём данных, запросы статистики и экспорта могут замедляться. Для небольшого объёма заявок это может быть приемлемо; при большом — архивация станет необходимостью позже.

## 6. Краткая сводка по краш-тестам

| Действие «неправильного» пользователя | Текущий результат | После исправлений |
|---------------------------------------|-------------------|--------------------|
| Подставить X-Telegram-User-Id: 386865879 | Доступ в админку | При заданном BOT_TOKEN проверяется initData — без валидного initData 401. |
| Открыть GET /api/bookings/BK-... без заголовка | Полные данные заявки | Только публичные поля (номер, статус, маршрут, дата, сумма); персональные данные — при авторизации владельца/диспетчера/админа. |
| POST cancel на чужую заявку | Заявка отменена | 403, отменить может только владелец или диспетчер/админ. |
| Отправить departure_date = "not-a-date" | 500 ValueError | 400 invalid_date_format. |
| Двойной клик «Забронировать» | Две заявки | На фронте — disable кнопки; опционально idempotency на бэке. |
| Два создания брони в одну секунду (одинаковый id) | Один 500 IntegrityError | Повтор с новым id. |
| Диспетчер без записи в dispatchers | 403 not_dispatcher | Без изменений. |
| Админ без ADMIN_IDS в env | 403 not_admin | Без изменений. |

После твоих ответов на вопросы из раздела 4 можно внедрить оставшиеся пункты и считать проект готовым к продакшену в задуманных границах.

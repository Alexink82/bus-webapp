# Глубокий анализ логики приложения и план UX/IA перестройки

Дата: 2026-03-12

## 1. Executive summary

Текущий проект уже покрывает основной путь пользователя (поиск → выбор рейса → оформление → успех), а также роли диспетчера/админа. Но есть ряд несоответствий профессиональному уровню:

1) Логические рассинхроны доступа/безопасности между ролями.
2) Разрозненная IA (информационная архитектура) и группировка меню между пассажирской, диспетчерской и админской частями.
3) Неполная унификация дизайн-системы (параллельные стили и различающиеся паттерны экранов).
4) Недостаточно единый feedback-pattern (alerts, ошибки, статусы, подтверждения).

Ниже — конкретные несоответствия и план «что добавить/как перестроить».

---

## 2. Найденные несоответствия в логике (backend + frontend)

### 2.1 Доступ диспетчера к деталям любой заявки (объектный контроль)
В `GET /api/bookings/{booking_id}` диспетчер получает полный доступ по факту роли диспетчера, без проверки, что заявка относится к маршрутам этого диспетчера.
- **Риск:** утечка персональных данных (телефон/пассажиры) между диспетчерами.
- **Что сделать:** добавить проверку `b.route_id in dispatcher_route_ids` для full-access ветки. *(В main уже исправлено.)*

### 2.2 WebSocket-уведомления о новых заявках рассылаются без фильтра маршрутов
- **Риск:** информационный шум и лишняя операционная нагрузка.
- **Что сделать:** хранить route-scopes соединения и фильтровать broadcast по `route_id`. *(В main уже реализовано.)*

### 2.3 В payment/create сумма берётся из клиента
- **Риск:** потенциальная подмена суммы при реальном эквайринге.
- **Что сделать:** брать amount только из `Booking.price_total`. *(В main уже исправлено.)*

### 2.4 IA-рассинхрон между ролями
Пассажирская часть использует нижнюю навигацию, диспетчер и админ — отдельные page-level tabs. Нет единого паттерна «где пользователь и в каком контуре находится».
- **Что сделать:** ввести role-shell (единый верхний слой с ролью/контекстом и быстрым переходом между контурами).

## 3. Что добавить для удобства использования

### 3.1 Пассажирский контур
1. Быстрые пресеты дат: Сегодня / Завтра / Выходные.
2. Память последнего маршрута + one-tap повтор последнего поиска.
3. Явный статус заявки в карточке с иконкой/цветом и подписью "что дальше".
4. Пустые состояния с конкретным действием (сменить дату/город).
5. Умный fallback телефона из профиля.

### 3.2 Диспетчерский контур
1. Вкладки: Новые / Мои в работе / Просроченные / Завершённые.
2. В карточке заявки: таймер SLA (сколько ждёт обработки).
3. Массовые операции: "взять N заявок", "сменить статус" для выделенных.
4. Фильтры: по маршруту, дате, статусу оплаты.

### 3.3 Админский контур
1. Dashboard из 4 KPI карточек (день/неделя/месяц переключатель).
2. Отдельный блок управления ролями с журналом изменений.
3. Логи: фильтр + поиск + экспорт.
4. Диагностика инфраструктуры: последний успешный старт, ошибки старта, health latency.

## 4. Как перестроить блоки на экранах (структура)

### 4.1 Главная (index)
Рекомендуемая структура: Header → Hero action-card (поиск) → Quick chips (последние направления + быстрые даты) → Results → Disclosure "Полное расписание" → Bottom bar. Блок "Актуальная информация" — ниже расписания, сворачиваемый.

### 4.2 Booking (step form)
Sticky stepper + sticky summary. Шаг 1: пассажиры (блоками карточек). Шаг 2: контакт + оплата. Bottom CTA bar. Явные ошибки рядом с полем + общий summary error вверху шага.

### 4.3 Dispatcher
Header + role badge → KPI strip (новые, в работе, просроченные) → Tabs + фильтры → Список карточек с action buttons одинаковой высоты.

### 4.4 Admin
Header + environment/status → KPI dashboard → Tabs: Analytics / Roles / Logs / Export. Формы в одном grid-ритме с одинаковыми вертикальными отступами.

## 5. Профессиональная IA группировка меню

**Нижнее меню (3–4 пункта):** Бронирование | Мои заявки | Профиль | FAQ (или Поддержка).
**Ролевые разделы:** Диспетчер и Админ — отдельные контуры с единым entry-point "Рабочая панель", внутри переключение ролей при наличии обеих.

---

## 6. UI/Design профессиональные доработки (roadmap)

1. Один источник токенов (цвет, spacing, radius, shadows, typography).
2. 8pt-grid enforcement: убрать случайные 10/14px там, где возможен 8/16/24.
3. Единый компонентный слой: Button, Card, Input, Tabs, Modal, Badge.
4. Контраст WCAG проверить автоматизированно (4.5:1).
5. Visual regression snapshots для ключевых экранов на CI.

## 7. Технические рекомендации (быстрые победы)

### P0 (срочно) — в main уже закрыто
- Объектный контроль доступа для диспетчера в GET booking details.
- Фильтрация WS broadcast по маршрутам.
- Серверная сумма оплаты из booking.

### P1 (спринт)
- Role-shell и единая навигационная модель.
- Единый feedback pattern (toast/alerts/status).
- KPI + фильтры для dispatcher/admin.

### P2 (дальше)
- Полная компонентная дизайн-система + дизайн-токены как единственный источник.
- UX-метрики: conversion (search→booking), drop-off по шагам, median time to complete.

---

## 8. Definition of Done для "профессионального уровня"

- Нет критичных расхождений role-based access.
- Навигация единообразна между ролями.
- Все интерактивы проходят 44x44 и keyboard/focus standards.
- Визуальные токены централизованы и соблюдается 8pt-grid.
- Ключевые пользовательские сценарии закрываются ≤ 4 действиями и с понятным feedback.

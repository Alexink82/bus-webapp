# План оптимизации — только пользовательская часть

**Исключено:** админка (admin), панель диспетчера (dispatcher).  
**В scope:** главная (index), бронирование (booking), профиль (profile), страница успеха (success), FAQ (faq).

**Цель:** LCP < 800 мс, повторные открытия < 200 мс для сценариев: главная → бронирование → успех, профиль (заявки, пассажиры, настройки), FAQ.

---

## A1. Vite multi-page — только пользовательские страницы

| # | Задача | Детали |
|---|--------|--------|
| A1.1 | Добавить Vite в проект | `package.json`: vite, скрипты `build` / `dev`. Корень сборки — `webapp`, `base: '/'`, `outDir: '../dist'` (или копирование в webapp при деплое). |
| A1.2 | Multi-page input только для user-страниц | В `rollupOptions.input` включить только: **index.html**, **booking.html**, **profile.html**, **success.html**, **faq.html**. Admin и dispatcher не трогать — остаются с текущими `<script src="...">` и отдаются как есть из того же webapp. |
| A1.3 | Entry-points по страницам | Один entry на страницу: `index-entry.js`, `booking-entry.js`, `profile-entry.js`, `success-entry.js`, `faq-entry.js`. Общие модули (api, auth, theme, i18n, app-modal, nav, consent, settings) — в общий chunk. |
| A1.4 | Карта скриптов (user) | **index:** maintenance, api, auth, theme, i18n, settings, nav, prefetch, app-modal, consent + инициализация маршрутов/формы. **booking:** api, auth, app-modal, i18n, settings, theme, consent, icons, booking; IMask/date-picker/input-masks — dynamic import при первом использовании. **profile:** maintenance, api, auth, theme, segmented-control, app-modal, nav, prefetch, consent, profile; IMask — при модалке паспорта; qrcode — при открытии модалки с QR. **success:** api, icons, auth, app-modal, i18n, settings, theme, nav + логика success (booking_id, cancel, haptic). **faq:** maintenance, api, auth, i18n, settings, theme, nav, prefetch + загрузка FAQ и поиск. |
| A1.5 | Подключение в HTML | В каждом из 5 HTML заменить набор `<script src="...">` на один `<script type="module" src="./js/...-entry.js">`. Telegram Web App скрипт оставить внешним. |
| A1.6 | CSS по страницам | В entry импортировать: main.css, telegram-theme.css, при необходимости animations.css; страничные — booking.css, date-picker.css (booking/profile). Один бандл CSS на страницу. |
| A1.7 | Сборка и деплой | После `npm run build` копировать `dist/*` в `webapp/` (или отдавать статику из dist). Admin и dispatcher по-прежнему читаются из webapp (их HTML не в rollup input — остаются как статические файлы с текущими скриптами). |

**Результат:** На index, booking, profile, success, faq — один JS + один CSS с хешами; тяжёлые библиотеки (IMask, qrcode) только при необходимости. Admin и dispatcher без изменений.

## A2. Клиентский кэш API — только пользовательские запросы

| # | Задача | Детали |
|---|--------|--------|
| A2.1 | Модуль кэша поверх api() | В `api.js` (или отдельный `cache.js`) обёртка с сохранением заголовков Telegram: по ключу (path + опционально user) проверять кэш в памяти, при попадании — возвращать данные; при промахе — вызов api(), запись в кэш, возврат. TTL настраиваемый (например 5 мин для dashboard, 10 мин для маршрутов). |
| A2.2 | Кэш для профиля | Загрузка заявок, пассажиров, профиля — через кэшированный вызов. После A3 — один запрос к `/api/user/dashboard` с кэшем. При открытии профиля: сначала рендер из кэша (если есть), затем фоновый запрос и обновление. |
| A2.3 | Кэш маршрутов | На index/booking запрос маршрутов — через кэш (TTL 5–10 мин). |
| A2.4 | Кэш FAQ | faq: fetch к `/api/faq?lang=...` перевести на api() и кэшировать по ключу url (или оставить сырой fetch с отдельным кэшем по url). |
| A2.5 | Инвалидация (только user) | После создания заявки (редирект на success) — инвалидировать `/api/user/bookings`, `/api/user/dashboard`. После добавления/редактирования/удаления пассажира — `/api/user/passengers`, `/api/user/dashboard`. После отмены/переноса заявки — те же ключи. Админка и диспетчерская в инвалидацию не входят. |

**Результат:** Повторное открытие главной и профиля — данные с экрана сразу из кэша, обновление в фоне.

## B4. Режим интерфейса (нормальный / экономный) — только для пользователей

| # | Задача | Детали |
|---|--------|--------|
| B4.1 | Настройка в профиле | В блоке «Настройки» переключатель «Режим интерфейса»: «Нормальный» / «Экономный». Значение в localStorage (`performanceMode` или `uiMode`). |
| B4.2 | Авто-определение при первом запуске | При отсутствии сохранённого значения: `navigator.hardwareConcurrency < 4` или `navigator.deviceMemory < 4` → по умолчанию «Экономный». Не перезаписывать, если пользователь уже выбирал вручную. |
| B4.3 | Уважение к системе | Если `prefers-reduced-motion: reduce` — не показывать тяжёлые анимации независимо от режима. |
| B4.4 | Поведение «Экономный» | Упростить: blur у модалок → fallback rgba; скелетон — без shimmer или статичный; анимация чекмарка на success — короче или выключена; QR в профиле — только по кнопке «Показать QR»; префетч — сократить или отключить. Без упоминания Chart/админки. |
| B4.5 | Единая точка проверки | Хелпер `isEconomyMode()` (localStorage + prefers-reduced-motion), использовать при загрузке QR, префетче, выборе анимаций. |

**Результат:** На слабых устройствах — быстрый интерфейс; пользователь может переключить вручную в настройках профиля.

## B5. Lazy-load тяжёлых библиотек — только пользовательские страницы

| # | Задача | Детали |
|---|--------|--------|
| B5.1 | IMask (booking, profile) | Подгружать только при первом фокусе/открытии полей с маской (дата, паспорт, телефон) или при открытии модалки редактирования пассажира. |
| B5.2 | qrcode (profile) | Подгружать только при открытии модалки «Подробнее» по заявке с QR. Либо перейти на npm `qrcode` с toCanvas/toDataURL после Vite. |
| B5.3 | date-picker | Если тяжёлый — подгружать при первом клике по полю даты на booking/profile. |

**Примечание:** Chart.js и lazy-load вкладок админки в этот план не входят.

## B6. Service Worker — статика пользовательских страниц

| # | Задача | Детали |
|---|--------|--------|
| B6.1 | Cache First для статики | Запросы к тому же origin, путь не начинается с `/api/` — Cache First (при первом запросе — fetch и в кэш, при повторном — из кэша). Не привязываться к `/webapp/` или фиксированному списку файлов; после Vite имена с хешами. |
| B6.2 | Версионирование кэша | CACHE_NAME с версией, при обновлении приложения — новая версия, старый кэш очищается. |
| B6.3 | API не кэшировать в SW | Запросы к `/api/` не кэшировать в Service Worker; кэш API только в логике приложения (A2). |

**Результат:** Повторные заходы на index, booking, profile, success, faq — JS/CSS/шрифты/картинки из кэша браузера.

- [ ] `npm run build` собирает index, booking, profile, success, faq; admin и dispatcher не входят в бандл и работают как раньше.
- [ ] На каждой user-странице один основной JS-chunk и при необходимости лениво подгружаемые модули.
- [ ] IMask, qrcode подгружаются только когда нужны (поля с маской, модалка с QR).

- [ ] Есть GET /api/user/dashboard; профиль использует один запрос и кэш.

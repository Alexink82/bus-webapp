# План оптимизации: мгновенный отклик + богатый интерфейс (без компромиссов)

**Дата:** 2026-03-16  
**Основа:** гибридный подход + Vite multi-page + клиентский кэш + режим «Экономный».

---

## Исходное состояние

- Многостраничный vanilla: 7 HTML-файлов, 17+ отдельных JS на страницу, CDN (IMask, qrcode.js, Chart.js).
- Скрипты подключаются по одному через `<script src>` без async/defer.
- sw.js есть; Redis в проде (rate-limit + кэш маршрутов); бандлинга нет (нет vite.config, нет dist/).

---

## Приоритет внедрения (по убыванию эффекта)

### Шаг 1. Vite multi-page (≈ +40 % скорости) — сделать первым

| # | Задача | Детали |
|---|--------|--------|
| 1.1 | Добавить Vite в проект | `package.json`: vite, скрипты `build` / `dev`. Опционально: отдельная папка `src/` или оставить `webapp/` как корень сборки. |
| 1.2 | Настроить multi-page build | Один `vite.config.js` с `build.rollupOptions.input` для всех HTML: index, booking, profile, success, faq, dispatcher, admin. Выход в `dist/` (или `webapp/dist/`). |
| 1.3 | Разделить entry-points по страницам | Для каждой страницы свой entry JS (например `index.js` → подключает только то, что нужно для index). Общие модули (api, auth, theme, i18n, app-modal) — в общий chunk. |
| 1.4 | Dynamic import тяжёлых библиотек | Chart.js — загружать только при переходе на вкладку «Статистика» в админке (`if (tab === 'statsPanel') { const Chart = await import('chart.js/auto'); }`). qrcode — только при открытии модалки «Подробнее» с QR. IMask — только на страницах с полями даты/паспорта (booking, profile модалка). |
| 1.5 | Подключение скриптов через Vite | В HTML заменить набор `<script src="...">` на один `<script type="module" src="/js/entry-profile.js">` (путь после сборки). Vite сам добавит defer/code-splitting и хеши в имена. |
| 1.6 | Сборка CSS | Один (или несколько) CSS на страницу: main + telegram-theme + страничные (booking, dispatcher, design-system). Минификация, при необходимости — разбиение на критичный + отложенный. |
| 1.7 | Проверка деплоя | После `npm run build` отдавать статику из `dist/`. Обновить конфиг Render/сервера: корень статики = `dist` (или `webapp/dist`). |

**Результат:** Один JS + один CSS на страницу, тяжёлые библиотеки только при необходимости. Парсинг и выполнение JS существенно быстрее.

### Шаг 2. Клиентский кэш API (мгновенный отклик со второго захода)

| # | Задача | Детали |
|---|--------|--------|
| 2.1 | Создать модуль кэша | Файл `cache.js` (или внутри api.js): `Map` в памяти + опционально запись в localStorage для ключевых URL. Функция `getWithCache(url, ttl)` (например TTL 5 мин): при попадании в кэш — сразу возвращать данные; иначе — вызов `api.get(url)`, сохранение в кэш, возврат. |
| 2.2 | Интегрировать в profile | Загрузка заявок, пассажиров, профиля — через `getWithCache`. При открытии профиля сначала рендер из кэша (если есть), затем фоновый запрос и обновление при изменении данных. |
| 2.3 | Интегрировать в админку | Список заявок, статистика, логи — через кэш с коротким TTL (1–2 мин) или только в памяти на время сессии. |
| 2.4 | Интегрировать маршруты | На index/booking маршруты запрашивать через кэш (уже есть серверный Redis; на клиенте дублировать кэш на 5–10 мин). |
| 2.5 | Инвалидация | При создании/отмене заявки, добавлении пассажира — сбросить соответствующие ключи кэша, чтобы следующий экран подтянул свежие данные. |

**Результат:** Повторное открытие профиля/главной — данные с экрана сразу из кэша, обновление в фоне.

### Шаг 3. Гибридный «Режим интерфейса» (нормальный / экономный)

| # | Задача | Детали |
|---|--------|--------|
| 3.1 | Добавить настройку в Профиль | В блоке «Настройки» переключатель «Режим интерфейса»: «Нормальный» (по умолчанию) / «Экономный». Значение в localStorage (`performanceMode` или `uiMode`). |
| 3.2 | Авто-определение при первом запуске | Один раз при отсутствии сохранённого значения: `isLowEnd = navigator.hardwareConcurrency < 4 || (navigator.deviceMemory && navigator.deviceMemory < 4)`. Сохранить `performanceMode: isLowEnd ? 'economy' : 'normal'`. Не перезаписывать, если пользователь уже выбирал вручную. |
| 3.3 | Уважение к системе | Если `prefers-reduced-motion: reduce` или уже включено «Уменьшить анимации» — считать это приоритетом: не показывать тяжёлые анимации независимо от режима. |
| 3.4 | Поведение в режиме «Экономный» | Отключить или упростить: blur у модалок (fallback rgba); анимация скелетона — статичная или без shimmer; анимация чекмарка на success — короче или отключена; Chart.js в админке не загружать (только цифры); QR — только по явной кнопке «Показать QR»; префетч вкладок — сократить или отключить. |
| 3.5 | Поведение в режиме «Нормальный» | Текущее: все анимации, blur, Chart.js при открытии «Статистика», QR в модалке, haptic, полный префетч. |
| 3.6 | Единая точка проверки | В коде один хелпер `isEconomyMode()` (читает localStorage + prefers-reduced-motion), использовать при принятии решений (загружать ли Chart, показывать ли blur, какую анимацию скелетона). |

**Результат:** На слабых устройствах по умолчанию — быстрый и лёгкий интерфейс; на нормальных — все эффекты; пользователь может переключить вручную.

### Шаг 4. Дополнительные меры (быстро и с большим эффектом)

| # | Задача | Детали |
|---|--------|--------|
| 4.1 | Обновить Service Worker | Стратегия Cache First для статики (JS, CSS, шрифты, изображения). Версионирование через hash в именах файлов (Vite даёт при build). При обновлении приложения — новая версия кэша, старый очищается. API не кэшировать в SW (оставить кэш в памяти/логике приложения). |
| 4.2 | Эндпоинт /api/user/dashboard | Один комбинированный ответ: заявки пользователя + сохранённые пассажиры + данные профиля (телефон и т.д.) + при необходимости сводка для блока статистики. Один запрос вместо водопада (bookings + passengers + profile). На фронте при открытии профиля — один вызов, заполнение всех блоков. |
| 4.3 | Lazy-load в админке | Вкладка «Все заявки»: данные запрашивать только при первом переключении на эту вкладку (не при загрузке admin.html). Вкладка «Статистика»: загрузка данных и Chart.js — при первом открытии вкладки. Остальные вкладки — по текущей логике. |
| 4.4 | Preload частых переходов | В `<head>` главной (index) добавить `<link rel="preload" href=".../profile.html">` и `<link rel="preload" href=".../booking.html">` (или preload ключевого JS/CSS для profile и booking после перехода на Vite). Чтобы браузер заранее подтягивал ресурсы при переходе пользователя в Профиль или в Бронирование. |
| 4.5 | ETag + stale-while-revalidate на бэкенде | Для GET-эндпоинтов с кэшем в Redis (маршруты, dashboard): отдавать ETag; при If-None-Match возвращать 304; опционально короткий Cache-Control с stale-while-revalidate, чтобы браузер мог отдать кэш и обновить в фоне. |

**Результат:** Меньше запросов при открытии профиля, быстрее повторные заходы за счёт кэша браузера и 304, админка не грузит лишнее до перехода на вкладку.

## Чек-лист перед закрытием плана

- [ ] `npm run build` собирает все страницы, статика отдаётся из dist.
- [ ] На каждой странице один основной JS-chunk (и при необходимости лениво подгружаемые).
- [ ] Chart.js, qrcode, IMask подгружаются только когда нужны.
- [ ] Профиль и главная при повторном открытии показывают данные из кэша мгновенно, затем обновляются в фоне.
- [ ] В настройках есть «Режим интерфейса»; при первом запуске выставляется авто; в экономном режиме нет blur, тяжёлых анимаций и автоматической загрузки Chart/qrcode.
- [ ] SW кэширует статику по Cache First с версионированием.
- [ ] Есть /api/user/dashboard (или аналог) и lazy-load данных/графиков в админке.
- [ ] Preload для profile и booking настроен (или эквивалент через Vite preload).

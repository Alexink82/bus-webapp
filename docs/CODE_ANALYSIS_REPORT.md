# Детальный анализ кода Bus Booking Web App

**Роли:** администратор, пассажир, диспетчер.  
**Проверено:** кнопки, переходы, API, связи между файлами, возможные ошибки в будущем.

---

## 1. Критические ошибки (могут ломать сценарии)

### 1.1 Сохранённые пассажиры без профиля (backend)

**Файл:** `backend/api/user.py` — `add_passenger`.

**Проблема:** `SavedPassenger` имеет `ForeignKey("user_profiles.user_id")`. При первом сохранении пассажира у пользователя может ещё не быть записи в `user_profiles`. В `add_passenger` профиль не создаётся, только добавляется `SavedPassenger` → при отсутствии `UserProfile` возможна ошибка целостности (IntegrityError).

**Рекомендация:** Перед `db.add(p)` обеспечить наличие профиля (аналог логики в `create_booking`):  
`select(UserProfile).where(UserProfile.user_id == user_id)` и при отсутствии — создать `UserProfile(user_id=user_id)` и `db.add(profile)` + `await db.flush()`.

---

### 1.2 Расчёт цены сегмента (backend)

**Файлы:** `backend/api/booking.py` — `_route_dict_for_calc`, `backend/services/price_calc.py` — `calculate_segment_price`.

**Проблема:** В `_route_dict_for_calc` задаётся `price_offset: base * i // max(len(stops), 1)`. В `calculate_segment_price` считается `base_price + (to_offset - from_offset)`. Для маршрута из 3 остановок получается переплата (например, сегмент 0→2 даёт сумму больше полного тарифа). Ожидаемая логика: либо сегмент = `(to_offset - from_offset)` при таких offset’ах, чтобы полный маршрут давал `base_price`, либо единая цена за сегмент без добавления base.

**Рекомендация:** Зафиксировать бизнес-правило (полный маршрут = base_price, сегмент — доля от него) и поправить формулу и/или расчёт offset’ов; добавить тесты на граничные маршруты.

---

### 1.3 Двойной commit в payment (избыточность)

**Файл:** `backend/api/payment.py` — `create_payment`, `payment_callback`.

**Проблема:** В обработчиках вызывается `await db.commit()`, при этом зависимость `get_db` по завершении запроса тоже делает `await session.commit()`. В итоге commit вызывается дважды (второй раз по сути пустой).

**Рекомендация:** Убрать явный `await db.commit()` в этих эндпоинтах и полагаться на commit в `get_db`, как в остальных роутерах (booking, user и т.д.), чтобы не путать контракт зависимостей.

---

## 2. Риски для пассажира

### 2.1 Бронирование без обязательных query-параметров

**Файл:** `webapp/js/booking.js` (начало).

**Проблема:** При открытии `booking.html` без `route_id`/`from`/`to`/`date`/`time` выполняется редирект на `index.html`. Если пользователь сохранит ссылку без параметров или откроет её из другого источника — попадёт на главную без явного сообщения («некорректная ссылка»).

**Рекомендация:** Кратко показывать сообщение вроде «Указан неполный маршрут» перед редиректом или на index показывать hint при `?error=invalid_booking_link`.

---

### 2.2 success.html — отмена заявки и порядок аргументов confirm

**Файл:** `webapp/success.html`.

**Проблема:** Вызов `showAppConfirm(msg, title)` корректен (message, title). Но при отсутствии скрипта модалки используется `confirm(msg)` — порядок тот же. Переменная `t` для перевода используется как `t('bookingId')`, `t('error')` и т.д., при этом `t` может быть не определена, если i18n не подгружен или не экспортирует `t` в глобальную область — возможна ошибка в консоли.

**Рекомендация:** Проверять наличие `typeof t === 'function'` перед вызовом (уже есть в части мест) везде, где используется `t`, либо гарантировать определение заглушки `window.t` в i18n.

---

### 2.3 Нет таймаута на fetch /api/routes в booking

**Файл:** `webapp/js/booking.js` — первый fetch.

**Проблема:** При долгом ответе (холодный старт Render, медленная сеть) запрос может висеть очень долго. Пользователь видит «Загрузка маршрутов…» без обратной связи.

**Рекомендация:** Добавить `AbortController` с таймаутом (например 15–20 с) и при срабатывании показывать тот же блок ошибки с «Повторить» и «Выбрать маршрут», как при сетевой ошибке.

---

### 2.4 Кэш старых скриптов после деплоя

**Файл:** `webapp/booking.html` (подключение скриптов).

**Проблема:** Уже добавлен `?v=2` для `booking.js`. Остальные страницы и скрипты (index, success, dispatcher, admin, api.js и т.д.) без версии — после деплоя пользователи могут получать старые версии из кэша.

**Рекомендация:** Ввести единую схему cache-busting (например, версия из package.json или дата сборки) для критичных JS/CSS на всех страницах или задать для статики webapp короткий `Cache-Control` (например `max-age=300`) на бэкенде.

---

## 3. Риски для диспетчера

### 3.1 Диспетчер берёт заявку другого диспетчера

**Файл:** `backend/api/dispatcher.py` — `take_booking`.

**Проблема:** Проверка `if b.dispatcher_id and b.dispatcher_id != dispatcher_id` даёт 409 только если заявка уже кем-то взята. Если два диспетчера одновременно нажмут «Взять», оба могут пройти проверку до commit и оба обновят запись (последний commit «выигрывает»).

**Рекомендация:** Использовать блокировку на уровне БД (SELECT FOR UPDATE) или оптимистичную проверку версии/updated_at при обновлении, чтобы исключить гонку.

---

### 3.2 set_booking_status — проверка «только свой заказ»

**Файл:** `backend/api/dispatcher.py` — `set_booking_status`.

**Проблема:** Проверка `b.dispatcher_id != dispatcher_id` не учитывает админа: админ не является диспетчером этой заявки, но по смыслу может менять статус. Сейчас админ не может вызвать этот эндпоинт для смены статуса (доступ только у диспетчера по `get_verified_telegram_user_id`).

**Рекомендация:** Либо явно разрешить админу менять статус (доп. проверка `is_admin(uid)` и разрешать без совпадения `dispatcher_id`), либо задокументировать, что смена статуса — только у взявшего диспетчера.

---

### 3.3 WebSocket — повторная отправка auth при реконнекте

**Файл:** `webapp/js/dispatcher.js` — `connectWs`, `ws.onopen`.

**Проблема:** При реконнекте после потери связи снова отправляется `{ type: 'auth', init_data: initData }`. Срок жизни initData в Telegram ограничен; при долгом простое старый initData может быть невалиден, и сервер закроет сокет с 4003.

**Рекомендация:** При закрытии с кодом 4003 (неавторизован) не делать автоматический reconnect с тем же initData, а показать сообщение «Сессия истекла, обновите страницу» и предложить перезагрузку.

---

### 3.4 Поиск заявки по номеру без проверки прав

**Файл:** `webapp/js/dispatcher.js` — поиск по `searchInput`; `GET /api/bookings/{id}`.

**Проблема:** Запрос идёт на публичный `GET /api/bookings/{id}`. В `api/booking.py` для доступа к полным данным проверяется владелец/админ/диспетчер. Диспетчер в списке имеет только свои маршруты (`get_dispatcher_route_ids`), но поиск по ID заявки возвращает заявку, если она входит в маршруты диспетчера. Для диспетчера с пустым списком маршрутов (все маршруты) — доступ ко всем заявкам по ID. Логика в целом корректна, но по одному ID диспетчер может увидеть заявку не своего маршрута, если бэкенд так разрешит. В текущей реализации доступ диспетчера к заявке определяется в get_booking через get_dispatcher_route_ids — т.е. заявка по route_id должна входить в список. Так что утечки нет, но стоит явно проверить, что в поиске диспетчер не получает лишних полей по заявкам «чужих» маршрутов. Сейчас GET /api/bookings/{id} возвращает разные наборы полей в зависимости от full_access — для диспетчера с доступом по маршруту возвращается полный объект. Ок.

**Рекомендация:** Оставить как есть, при добавлении новых полей в ответ — проверять, что диспетчер видит только то, что нужно для работы.

## 4. Риски для администратора

### 4.1 admin.html — дублирование логики api и заголовков

**Файл:** `webapp/admin.html` — локальная функция `api(path)` и ручная подстановка заголовков.

**Проблема:** Логика запросов и заголовков (X-Telegram-User-Id, X-Telegram-Init-Data) продублирована относительно `api.js`. При изменении формата авторизации (новые заголовки, куки) нужно не забыть обновить и admin.

**Рекомендация:** Использовать общий `window.api` из `api.js` на странице админки вместо локальной обёртки, чтобы авторизация была в одном месте.

---

### 4.2 Экспорт CSV — тип ответа

**Файл:** `webapp/admin.html` — кнопка «Экспорт CSV».

**Проблема:** Используется `r.blob()`. Бэкенд отдаёт `text/csv`; для скачивания файла с именем это нормально. Стоит убедиться, что при больших объёмах StreamingResponse не буферизует всё в память (в текущей реализации передаётся `iter([output.getvalue()])` — весь CSV в памяти один раз).

**Рекомендация:** Для очень больших выборок рассмотреть потоковую отдачу по кускам; для текущих объёмов оставить как есть.

---

### 4.3 load_roles после add_admin — сессия

**Файл:** `backend/api/admin.py` — `add_admin`.

**Проблема:** После `await db.commit()` вызывается `await load_roles(db)`. В `load_roles` делается только чтение из BotRole. После commit сессия в допустимом состоянии для нового запроса. Но `load_roles` кэширует роли в глобальные переменные; при нескольких воркерах/процессах другие инстансы не узнают о новом админе до перезапуска.

**Рекомендация:** Задокументировать: «После добавления админа новый админ должен обновить страницу; при нескольких воркерах может потребоваться перезапуск воркеров для распространения ролей». При необходимости — вынести кэш ролей в Redis и инвалидировать при изменении.

## 5. Связи между файлами и соглашения

### 5.1 Глобальные переменные (window)

**Где задаются:** `api.js` — BASE_URL, api, getTelegramUserId, getTelegramInitData, escapeHtml, userFriendlyMessage, ERROR_MESSAGES; `app-modal.js` — showAppAlert, showAppConfirm, errorToMessage; `auth.js` — только side-effect (localStorage).

**Проблема:** Страницы, подключающие только часть скриптов (например, success без app-modal), при вызове showAppAlert получат fallback на `alert`. Если на странице нет api.js, но есть код, вызывающий api() или BASE_URL — будет ошибка.

**Рекомендация:** В шаблоне или в одном общем скрипте проверять наличие обязательных глобальных (BASE_URL, api) перед выполнением логики страницы; для модалок оставить текущий fallback на alert/confirm.

---

### 5.2 Порядок скриптов в booking.html

**Файл:** `webapp/booking.html`.

**Порядок:** api.js, auth.js, imask, input-masks, passport-config, phone-config, date-picker, app-modal, i18n, settings, theme, consent, booking.js.

**Проблема:** booking.js ожидает BASE_URL, api, getTelegramUserId, getPassengerDiscountLabel, dobToIso, datePickerIsoToDisplay, isoToDob, passportFormatDisplay, passportFormatInput, passportCleanForApi, passportValidate, getPassportCountry, PASSPORT_TOP_COUNTRIES, IMask и др. Часть из них — из других скриптов (input-masks, passport-config, phone-config, date-picker). При изменении порядка или переименовании файлов возможны ReferenceError.

**Рекомендация:** Держать список зависимостей booking.js в комментарии в начале файла или в README; при добавлении новых скриптов проверять порядок загрузки.

---

### 5.3 Навигация и вкладки (nav.js)

**Файл:** `webapp/js/nav.js`.

**Проблема:** Вкладки «Диспетчер» и «Админ» подставляются только если есть `#tabBar` и ссылка на faq.html. На страницах без tabBar (dispatcher.html, admin.html, booking.html, success.html) навигация не обновляется — это ожидаемо. Но если переименовать или удалить faq.html, insertBefore сломается.

**Рекомендация:** Проверять наличие `faqTab` перед вставкой; при рефакторинге навигации обновить селектор.

## 6. Безопасность и валидация

### 6.1 CreateBookingIn — лишние поля

**Файл:** `backend/api/booking.py` — `CreateBookingIn`.

**Проблема:** Фронт может отправлять `save_passengers_to_profile`. В Pydantic по умолчанию extra-поля можно игнорировать (в зависимости от модели). В коде нет поля `save_passengers_to_profile` — оно просто не используется на бэке; сохранение пассажиров делается отдельными вызовами с фронта после создания заявки. Это не ошибка, но при включении strict-валидации запросы с лишними полями могут начать падать.

**Рекомендация:** Либо добавить в модель опциональное поле `save_passengers_to_profile: bool = False`, либо явно задать `model_config = ConfigDict(extra='ignore')`, чтобы контракт был предсказуемым.

---

### 6.2 CORS и credentials

**Файл:** `backend/main.py` — CORSMiddleware.

**Проблема:** `allow_credentials=False` и `allow_origins=settings.cors_origins` (при пустом ALLOWED_ORIGINS — `["*"]`). При `credentials=False` браузер не отправляет куки с cross-origin запросами; для текущей схемы (только заголовки X-Telegram-*) этого достаточно. Если позже появятся куки для сессий, нужно будет пересмотреть CORS и origins.

**Рекомендация:** Оставить как есть; при добавлении cookie-сессий — ограничить `allow_origins` конкретными доменами и при необходимости включить credentials.

---

## 7. Потенциальные сбои в будущем

### 7.1 Рост таблицы log_entries

**Файл:** `backend/logging_config.py` — `log_action` пишет в `LogEntry`; админка читает с limit.

**Проблема:** При большом потоке заявок и действиях диспетчеров/админов таблица будет быстро расти. Нет ротации и очистки.

**Рекомендация:** Периодическая задача (cron/фоновый воркер) удалять или архивировать записи старше N дней; либо писать логи только в файл/внешний сервис, а в БД хранить только последние N записей для админки.

---

### 7.2 Rate limit — in-memory

**Файл:** `backend/main.py` — `_rate_limit_store`.

**Проблема:** Словарь в памяти процесса. При нескольких воркерах Render каждый воркер ведёт свой счётчик — лимит по факту умножается на число воркеров. При перезапуске счётчики обнуляются.

**Рекомендация:** Для строгого лимита использовать Redis или аналог; иначе задокументировать поведение при нескольких воркерах.

---

### 7.3 Константы маршрутов (ROUTES) — рассинхрон с ботом

**Файлы:** `backend/core/constants.py`, бот (отдельный репозиторий).

**Проблема:** Маршруты захардкожены в constants. При изменении в боте нужно не забыть обновить webapp и наоборот — возможна рассинхронизация тарифов и остановок.

**Рекомендация:** Вынести ROUTES в общий пакет/репозиторий или хранить в БД с версионированием и подгружать при старте; при изменении — один источник правды.

---

### 7.4 Отсутствие индексов (миграции)

**Файл:** Модели и миграции Alembic.

**Проблема:** Частые запросы: `Booking.route_id`, `Booking.date`, `Booking.contact_tg_id`, `Booking.dispatcher_id`, `Booking.created_at`, `LogEntry.timestamp`, `SavedPassenger.user_id`. При росте данных отсутствие подходящих индексов замедлит выборки.

**Рекомендация:** Проверить существующие индексы и при необходимости добавить миграции под фильтры и сортировки (например, по date, route_id, contact_tg_id, created_at для bookings).

# Полный список работ по панели диспетчера

**Цели:** минимум кликов до действия, всё важное на одном экране, desktop-first с откатом на мобильный.

---

## Фаза 0. Подготовка (рефакторинг и база)

| # | Задача | Детали |
|---|--------|--------|
| 0.1 | Единая функция рендера карточки | Вынести `renderNewCard` / рендер карточки «В работе» в одну переиспользуемую функцию (или два варианта: компактная для списка, полная для виджета). Передавать данные заявки и тип (new / active). |
| 0.2 | Данные заявки в API | Убедиться, что GET `/api/bookings/{id}` для диспетчера возвращает `contact_phone`, `passengers`. При необходимости добавить в ответ `contact_tg_id` для ссылки «Написать в Telegram» в модалке контакта. |
| 0.3 | WebSocket: обновление при смене статуса | Сейчас WS шлёт только `new_booking`. Добавить рассылку при смене статуса заявки (take, status), чтобы у всех подключённых диспетчеров обновлялись списки и виджет активной заявки. |

## Фаза 1. Desktop: трёхколоночная верстка и топ-бар

| # | Задача | Детали |
|---|--------|--------|
| 1.1 | Топ-бар фиксированный (60 px) | Поиск (глобальный) всегда виден; счётчик просроченных за смену (SLA); при необходимости — текущее время. На мобильном топ-бар остаётся, но компактнее. |
| 1.2 | Контейнер 3 колонки (≥1024 px) | `.dispatcher-container`: grid `280px 1fr 340px`, `height: calc(100dvh - 60px)`, отступы. Левая колонка — фильтры + статистика смены; центр — табы + overdue + карточки; правая — «Моя активная заявка» + пассажиры + кнопки. |
| 1.3 | Левая колонка (sidebar) | Фильтры (маршрут, дата, оплата), кнопка «Применить», блок «Статистика смены» (за сегодня: кол-во заявок, сумма, просрочено >15 мин). Без фильтра по водителю. |
| 1.4 | Центральная колонка (main) | Segmented-табы: «Новые» / «В работе». Таб «Поиск» убрать — строка поиска перенесена в топ-бар. Таб «Статистика» убрать или заменить на мини-дашборд (цифры уже в левой колонке). Overdue-блок вверху таба «Новые». Сетка карточек на ПК: 1–2 колонки (grid). |
| 1.5 | Правая колонка (right panel) | Постоянный блок «Моя активная заявка»: если есть взятая текущим диспетчером заявка — показывать booking_id, маршрут, время отправления, таймер до рейса, список пассажиров, кнопки «Отправить билет», «Связаться», «Перенести», «Отменить». Если нет — заглушка «Нет активной заявки» или «Возьмите заявку из списка». Обновление по WebSocket при take/status. |
| 1.6 | Мобильный fallback (<1024 px) | Одна колонка: табы сверху, под ними overdue (в «Новых»), список карточек в одну колонку. Левая и правая панели скрыты: фильтры — в кнопку/таб «Фильтры» или выпадающую панель; «Моя активная заявка» — отдельный таб или плавающая кнопка, по нажатию — модалка с тем же содержимым. |

## Фаза 2. Виджет «Моя активная заявка» и контакт

| # | Задача | Детали |
|---|--------|--------|
| 2.1 | Логика «текущая активная заявка» | Определять заявку, взятую текущим диспетчером (dispatcher_id = uid), со статусом active. При загрузке и по WebSocket обновлять единственную «активную» для отображения в правой панели (и в мобильной модалке). |
| 2.2 | Содержимое виджета | booking_id, маршрут, дата/время отправления, таймер до рейса (обратный отсчёт), список пассажиров (ФИО, при необходимости паспорт), кнопки: «Отправить билет», «Связаться», «Перенести», «Отменить с причиной». |
| 2.3 | Модалка «Контакт пассажира» | По клику на контакт (телефон/иконка) в карточке или в виджете — модалка: телефон (кликабельный tel:), ссылка «Написать в Telegram» (если есть contact_tg_id — tg://user?id= или веб-вариант), при возможности — краткая история поездок по этому контакту (опционально, если есть эндпоинт). |
| 2.4 | Кликабельный контакт в карточках | В каждой карточке (Новые / В работе) контакт (телефон или «Связаться») ведёт в ту же модалку контакта (данные подгружать по GET /api/bookings/:id при открытии модалки). |

## Фаза 3. WebSocket и real-time

| # | Задача | Детали |
|---|--------|--------|
| 3.1 | Рассылка при смене статуса / take | В backend при POST take и POST status слать через WebSocket manager сообщение диспетчерам (по route_id или всем): тип `status_changed` или `booking_updated` с booking_id и статусом. На фронте в dispatcher.js по этому сообщению вызывать loadNew(), loadActive() и обновлять виджет активной заявки (если затронута текущая). |
| 3.2 | Обновление виджета и счётчиков | При получении WS: перезапросить список в текущем табе, обновить правую панель (если booking_id совпадает с активной заявкой), обновить счётчик просроченных в топ-баре и в левой колонке. |

## Фаза 4. SLA-таймеры и быстрые действия

| # | Задача | Детали |
|---|--------|--------|
| 4.1 | SLA-таймер на карточке | Для новых заявок: время с created_at. Визуализация: прогресс-бар или цветовая полоска (зелёный 0–10 мин, жёлтый 10–15, красный >15). То же для overdue-блока. |
| 4.2 | Быстрые действия в карточке | Три кнопки (или иконки): «Отправить ссылку оплаты», «Отправить билет» (QR/статус ticket_sent), «Отменить с причиной» (модалка ввода причины, затем POST status cancelled). При необходимости — проверка прав и состояний (например, ссылку оплаты только для определённого статуса). |
| 4.3 | Кнопки в виджете «Моя активная заявка» | Те же действия: отправить билет, связаться (модалка контакта), перенести (если есть эндпоинт/модалка переноса), отменить с причиной. |

## Фаза 5. Удобства для ПК

| # | Задача | Детали |
|---|--------|--------|
| 5.1 | Размеры на ПК | На ≥1024 px: карточки крупнее (padding 20 px, min-height 140 px), шрифт 16 px; бейджи 15 px; кнопки min-height 52 px, иконки 24 px. |
| 5.2 | Клавиатурные сокращения | Ctrl+K — фокус в поиск; Ctrl+1 / Ctrl+2 — переключение табов «Новые» / «В работе»; Enter — взять первую заявку в работе (или первую новую); при желании V — взять, S — отправить билет, C — отменить (с подтверждением). | 5.4 | Тёмная тема по умолчанию на ПК | Опционально: при ширине ≥1024 px подставлять тёмную тему, если пользователь не выбрал светлую (или учитывать системную prefers-color-scheme). |

## Порядок реализации (рекомендуемый)

1. **Фаза 0** — рефакторинг карточек, проверка API, WebSocket status_changed.  
2. **Фаза 1** — топ-бар, трёхколоночная сетка, левая/правая колонки, мобильный fallback.  
3. **Фаза 2** — виджет «Моя активная заявка», модалка контакта, кликабельный контакт в карточках.  
4. **Фаза 3** — доработка WebSocket и обновление списков/виджета в real-time.  
5. **Фаза 4** — SLA-таймеры, быстрые кнопки в карточке и в виджете.  
6. **Фаза 5** — размеры для ПК, горячие клавиши, двойной клик, тема.  
7. **Фаза 6–7** — массовые действия, экспорт, опционально Drag & Drop и карта.

## 2) Карта взаимосвязей между файлами

- `webapp/js/api.js` формирует заголовки `X-Telegram-User-Id` и `X-Telegram-Init-Data`, а backend проверяет их в `backend/api/auth_deps.py`.
- `backend/main.py` подключает все роутеры (`booking`, `user`, `dispatcher`, `admin`, `payment`, `faq`, `routes`, `websocket`) и раздаёт `webapp/` как статический фронтенд.
- `backend/api/booking.py` использует:
  - маршруты и цены из `backend/core/constants.py`,
  - валидацию из `backend/services/validators.py`,
  - роли из `backend/services/roles.py`,
  - уведомления из `backend/services/notification.py`.
- `backend/api/dispatcher.py` и `backend/api/admin.py` делят общую модель `Booking` (`backend/models.py`) и общий слой ролей (`backend/services/roles.py`).
- `backend/api/payment.py` обновляет ту же заявку `Booking` и таблицу `WebPayTransaction`.
- `backend/api/websocket.py` зависит от Telegram-валидации (`backend/services/telegram_auth.py`) и диспетчерских прав (`backend/services/roles.py`).

## 3) Главные сильные стороны

1. **Единая ролевая модель** (env + БД роли) с повторным использованием в API.
2. **Защита Telegram initData** при наличии `BOT_TOKEN`.
3. **Базовая защита от спама** через rate limit middleware.
4. **Покрыт минимальный smoke-набор API-тестов** для health/routes/ошибок авторизации.
5. **Аккуратная деградация**: при ошибках нотификаций/парсеров процесс не падает.

## 4) Критичные/высокие риски (найдено)

### [CRITICAL] Утечка деталей заявки для любого диспетчера
В `GET /api/bookings/{booking_id}` проверяется факт «пользователь — диспетчер», но не проверяется, что заявка относится к маршрутам этого диспетчера.

**Риск:** диспетчер A может получить детальные данные заявки (включая телефон и пассажиров) диспетчера B по `booking_id`.

**Рекомендация:** в `booking.py` для ветки dispatcher добавить проверку `b.route_id in allowed_route_ids`.

### [HIGH] WebSocket broadcast отправляет новые заявки всем диспетчерам
`broadcast_new_booking` отправляет событие всем подключённым, параметр `route_id` не используется для фильтрации.

**Риск:** лишние уведомления, шум, потенциальная утечка операционных данных по чужим направлениям.

**Рекомендация:** хранить route-scopes соединений и фильтровать рассылку по `route_id`.

### [HIGH] Сумма платежа доверяется клиенту
В `/api/payment/create` сумма (`amount`) берётся из тела запроса без серверной сверки с `Booking.price_total`.

**Риск:** клиент может оплатить меньшую сумму при реальном платёжном провайдере.

**Рекомендация:** игнорировать входной `amount`, всегда вычислять/читать сумму на сервере из брони.

### [MEDIUM] Тестовое покрытие не ловит часть регрессий по правам
Существующие тесты хорошо покрывают smoke и 401-сценарии, но слабо покрывают матрицу прав доступа «владелец/диспетчер/админ/чужой пользователь».

**Рекомендация:** добавить параметризованные API-тесты для access-control матрицы (особенно `GET booking details`, `cancel`, dispatcher operations).

## 5) Потенциальные будущие ошибки

1. **Рост памяти в in-memory rate-limit** при большом числе уникальных IP (особенно под прокси).
2. **Строковые даты/время в `Booking`** усложняют строгую аналитику, timezone и индексацию.
3. **Смешанная модель статусов оплаты/заказа** может привести к рассинхронизации (`paid_at` vs `status`).
4. **Отсутствие идемпотентности create_booking на backend** при сетевых ретраях клиента.
5. **Ограниченный health-check** без проверки доступности БД/критичных зависимостей.

---

## 6) Современные подходы (что внедрить)

1. **Policy-based authorization (RBAC/ABAC)**
   - Вынести правила доступа в отдельный слой (`can_view_booking`, `can_cancel_booking`, ...).
   - Уменьшит дублирование и вероятность расхождения правил в разных эндпоинтах.

2. **Idempotency-Key для write-операций**
   - Для `/api/bookings`, `/api/payment/create`, `/api/bookings/{id}/cancel`.

3. **Typed DTO + доменная схема статусов**
   - Централизовать переходы статусов через state-machine (например, allowed transitions map).

4. **Outbox pattern для уведомлений/WebSocket**
   - Чтобы нотификации были надёжны и повторяемы при временных сбоях.

5. **Наблюдаемость production-уровня**
   - Метрики (Prometheus/OpenTelemetry), request-id, structured logging, SLO/error budget.

6. **Безопасность веб-слоя**
   - CSP, security headers (HSTS, X-Content-Type-Options, Referrer-Policy), ужесточение CORS в проде.

## 7) Ролевые краш-тесты: резюме

### Пассажир
- Некорректные дата/телефон/пассажиры корректно отсекаются в API.
- Отмена защищена проверкой владельца/ролей.
- Основной риск: если `booking_id` угадан, детализация зависит от проверок доступа (см. critical issue).

### Диспетчер
- Базовый контур работает: список, take, status, websocket auth.
- Главный риск: избыточный доступ к чужим заявкам через `GET /api/bookings/{id}`.

### Админ
- Рабочие endpoints статистики/логов/экспорта.
- Риск смещён в сторону операционных практик (аудит-логи, контроль изменений ролей, мониторинг).

## 10) Вывод

Проект находится в рабочем состоянии и уже имеет хорошую основу для ролей, Telegram-интеграции и пользовательских сценариев. Основной технический долг сейчас — **точность авторизации на уровне объекта (booking-level access control)**, **маршрутная изоляция realtime-уведомлений** и **серверная верификация платёжной суммы**. Закрытие этих трёх пунктов даст максимальный выигрыш по безопасности и эксплуатационной надёжности.
